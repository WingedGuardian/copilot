EXECUTIVE CO-PILOT v1.0
Final Technical Specification & Build Document
Foundation: HKUDS/nanobot (Extended via Native Skills)
Interface: Regular WhatsApp (Baileys Library)
Architecture: Split - Windows 11 5070ti (GPU Inference) + Incus Container (Orchestration) On Proxmox VM (both on LAN)
Status: Ready for Implementation
1. EXECUTIVE SUMMARY
An autonomous executive assistant extending nanobot via native and non native features with task persistence (multi-day workflows) and intelligent routing that fails cheaply. Uses LM Studio on Windows 5070 for intelligent routing and trivial queries, Cloud LLMs for complex work, with automatic fallback when the gaming PC is offline.  
Infrastructure Split:

    Windows PC (5070ti GPU): Runs LM Studio serving Phi-4/Llama 3.2 via OpenAI-compatible API on 192.168.50.100:1234
    Incus Container (Ubuntu 24.04): Runs nanobot, QDrant, Redis, and all orchestration logic (ip: 192.168.50.123) Running on Proxmox VM

Core Innovation: The Sentry Routing Protocol with overflow protection—inputs >4096 characters or containing images bypass local SLM immediately to prevent context window crashes, while structured JSON output ensures reliable intent classification.

3. CORE COMPONENTS
3.1 Router (Sentry)
Location: skills/sentry_router/router.py
Pre-flight Checks:

    Input >4096 characters → Cloud
    Image attachment → Cloud
    Context + input >3500 tokens → Cloud

Local Attempt (LM Studio):

    Timeout: 5 seconds
    Confidence threshold: 0.70
    Model: llama-3.2-3b-instruct (4k context)
    Temperature: 0.1
    Structured JSON output required

Cloud Routing (OpenRouter/Venice):

    Primary: Aggregator API (OpenRouter or Venice)
    Secondary: MiniMax API (specific use cases)
    Models available: GPT-4o, Claude 3.5 Sonnet, MiniMax-Text-01, or latest available
    Automatic failover between providers if primary down

Failover Chain:

    LM Studio (local) - if timeout/failure →
    OpenRouter/Venice (cloud) - if failure →
    MiniMax (cloud backup)

4. DATA SCHEMA
SQLite (data/sqlite/copilot.db):
sql
Copy

CREATE TABLE tasks (
    id TEXT PRIMARY KEY,
    description TEXT,
    status TEXT CHECK(status IN ('pending', 'active', 'awaiting_approval', 'completed', 'failed')),
    route_attempted TEXT,
    created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
    completed_at TIMESTAMP,
    thread_id TEXT,
    error_log TEXT
);

CREATE TABLE routing_log (
    timestamp TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
    input_length INTEGER,
    routed_to TEXT,
    provider TEXT,  -- 'lmstudio', 'openrouter', 'venice', 'minimax'
    model_used TEXT,
    confidence REAL,
    success BOOLEAN,
    latency_ms INTEGER,
    failure_reason TEXT,
    cost_usd REAL
);

CREATE TABLE approval_rules (
    id INTEGER PRIMARY KEY,
    pattern TEXT,
    condition TEXT,
    action TEXT CHECK(action IN ('auto_approve', 'require_approval')),
    hit_count INTEGER DEFAULT 0,
    created_by TEXT  -- 'user' or 'system'
);

CREATE TABLE cost_tracking (
    timestamp TIMESTAMP,
    provider TEXT,
    model TEXT,
    tokens_input INTEGER,
    tokens_output INTEGER,
    cost_usd REAL,
    task_type TEXT
);

8. ANTICIPATED CHALLENGES & MITIGATIONS
Table
Copy
Challenge	Risk	Mitigation
LM Studio Windows PC offline	High (gaming, sleep mode)	Implement 5s timeout, immediate failover to Clooud LLM, queue non-urgent tasks
4096 Context Hard Limit	Medium (crashes if exceeded)	Strict pre-flight check, never send >3500 tokens to Windows PC
Structured Output Parsing Failures	Medium (3B model hallucinates JSON)	Try/catch with regex fallback, route to cloud on parse error
Image Handling	Low (user expects vision)	Immediate cloud route, implement vision in V2  
Windows Firewall Changes	Low (IP changes, blocks)	Static DHCP reservation on router, ping monitoring in health check
