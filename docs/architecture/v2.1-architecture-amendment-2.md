## Critical Bugs Found in Current Codebase

These exist TODAY in V1 and will compound in V2.1 if not fixed.

### Bug 1: Compaction Uses Wrong Model Context Window
**File**: `loop.py:725-733`, `extended.py:143-149`
**Impact**: CRITICAL

`needs_continuation()` checks token budget against `self.model`, which is always the local model string (Qwen 30B, ~8K context). But the router may have sent the actual call to Claude Sonnet (1M context). Result: compaction triggers at ~5,700 tokens even when the LLM has 200K+ available. Every moderately long conversation drops to 4 messages of history.

**Fix**: Compaction must use the actual routed model's context window.

### Bug 2: Heuristic Rule 7 Causes Silent Cost Drift
**File**: `heuristics.py`, rule 7
**Impact**: HIGH

Token count > 3,000 → route to `fast` (cloud). As conversations grow, they inevitably cross this threshold. Conversations silently migrate from local (free) to cloud (paid) with no user visibility.

**Fix**: Addressed by router redesign (see Decision #30 below).

### Bug 3: Router Ignores 3 of Its Own Classification Inputs
**File**: `router.py:183-202`
**Impact**: MEDIUM

`RouterProvider.chat()` never passes `tool_names`, `has_tool_calls`, or `lessons` to `classify()`. Rules 4, 8, and 10 in `heuristics.py` are dead code.

**Fix**: Addressed by router redesign (see Decision #30 below).

### Bug 4: Memory Consolidation Gets Misrouted
**File**: `loop.py:964-1032`
**Impact**: MEDIUM

`_consolidate_memory()` calls `self.provider.chat()` using `self.model`. The router classifies the consolidation prompt as simple text and routes to local. A local model doing memory extraction may produce lower-quality results.

**Fix**: Route consolidation explicitly to a mid-tier model, not through the heuristic classifier.

### Bug 5: TaskWorker execute_fn Signature Mismatch
**File**: `tasks/worker.py:127-133`
**Impact**: LOW (functional accident)

Worker calls `execute_fn(step.description, task.session_key, "cli", step.tool_type, step.recommended_model)`. But `process_direct()` signature is `(content, session_key, channel, chat_id, model)`. `tool_type` gets passed as `chat_id`. Works by accident but semantically wrong.

**Fix**: Correct the parameter mapping.

## Decisions Made (this session)

### Decision #30: Router Redesign — Simplify to Default Model + Escalation [IMPLEMENTED 2026-02-20]

**REPLACES**: The entire heuristic classification system (`heuristics.py`), silent model switching, and multi-tier routing for conversation.

**Current system**: 11-rule heuristic classifier picks local/fast/big per-message. Silent switching. Causes bugs #1, #2, #3.

**New system**:
- **Default model**: One cheap cloud model for all conversation (Haiku 4.5 at $1/$5 per MTok, or Gemini Flash at $0.50/$3.00 per MTok). Configurable in `CopilotConfig.default_conversation_model`.
- **Self-escalation**: If the default model responds with `[ESCALATE]`, retry with a stronger model (`CopilotConfig.escalation_model`, default Sonnet 4.5). One hop only.
- **Manual override**: `/use` command or natural language ("switch to Sonnet", "use Opus for this"). Nanobot's LLM handles intent detection naturally.
- **Local models**: Opt-in only via `/use local`. Not in the default path. Escalation instruction still injected for local.
- **Failover chain**: Preserved. If the default provider is down, fall back to an alternative provider serving the same model class.
- **Circuit breaker**: Preserved. Provider-level resilience unchanged.
- **Cost logging**: Preserved. Every call logged regardless of routing.

**What gets deleted**: `heuristics.py` (the classify function). `RouterProvider` becomes `ConversationProvider` — much simpler. No per-message classification. No silent tier switching.

**What this fixes**:
- Bug #1: One known model → correct compaction window
- Bug #2: No automatic cost drift — model is explicit
- Bug #3: No heuristic inputs to wire (no heuristics)
- Context continuity: Same model across conversation = consistent interpretation
- Predictable costs: User knows exactly which model they're paying for

**What this costs**: Slightly higher cost than routing simple messages to local. But the local model wasn't free anyway (LM Studio GPU time), and the bugs it introduced cost more in lost context and debugging time.

### Decision #31: Context Bridging on Model Switch

When the active model changes (manual `/use`, escalation, or new session):
- Inject a system note: `"Note: You are continuing a conversation previously handled by {previous_model}. Review the history carefully for context."`
- ~20 tokens. Dramatically helps the new model orient itself.
- Track `last_model_used` in session metadata (already exists but unused). Use it to detect switches.

If switching from a large-context model to a smaller one, compaction may fire. The bridging note warns the model that history may be truncated.

### Decision #32: Single Orchestrator at a Time

**REPLACES**: All multi-orchestrator infrastructure (global semaphore, TOCTOU budget races, `_running_orchestrators` dict).

- One task executes at a time. Tasks queue by priority. Next task starts when current finishes.
- Workers within a single task can still parallelize (up to `max_parallel_workers`, default 3-4).
- Eliminates: SQLite write contention, event loop starvation, budget TOCTOU, global semaphore complexity.
- `TaskWorker` tracks `_current_orchestrator: asyncio.Task | None` (not a dict).
- Multi-orchestrator deferred to V2.3+ if empirically needed.

**Task parking**: New `parked` status. User says "park this, I need task B." Orchestrator saves full workflow state (same as crash recovery), terminates. Next queued task starts. Resume later via `/resume <id>` — orchestrator relaunches, reads recovery briefing, picks up where it left off.

### Decision #33: Cross-Model Deliverable Review

Before the orchestrator delivers a draft to the user, a **different model** reviews it.

- **Reviewer model**: Configurable via `CopilotConfig.reviewer_model`. Defaults to a cost-effective capable model (Gemini Flash, MiniMax M2.5, or GLM-5). NOT the same as the orchestrator model.
- **Review prompt**: "Original request: {task}. Deliverable: {draft}. Score 1-5 on completeness, accuracy, relevance. List specific gaps. Return JSON."
- **Automated iteration**: If score < 4 on any dimension, orchestrator gets reviewer feedback, re-plans, re-executes. Max 2 automated iterations.
- **User sees draft only after**: Both orchestrator and reviewer agree it's ready, OR 2 automated iterations are exhausted (in which case, user is told what the reviewer flagged).
- **Cost**: ~$0.01-0.10 per review call. Max ~$0.20 for the review portion of a task.
- **The orchestrator CANNOT override the reviewer model.** The whole point is a different perspective. If the deliverable is too large for the reviewer's context, the orchestrator summarizes first.

### Decision #34: Revised Task Budget

**REPLACES**: $0.50 default budget (which can't even cover one Opus workflow design call).

- **Default budget**: $2.00 per task. Covers a typical 2-3 worker task with one retry cycle.
- **Soft limit with user approval**: At 80% budget, orchestrator notes it internally. At 100%, instead of aborting:
  ```
  [Task #3 — Orchestrator] I've used the $2.00 budget. Here's what I have
  so far: {partial result}. Want me to continue? Estimated additional
  cost: ~$X for remaining work.
  ```
- **User approval resets/extends budget.** User can approve continuation, at which point cost estimate + current spend are shown.
- **All orchestrator calls count toward budget**: Workflow design, diagnosis, re-planning, review, retrospective. Nothing is free.
- **Per-task budget adjustable**: Orchestrator can request higher budget during bootstrap confirmation. User can set via `create_task(budget=5.00)`.

### Decision #35: No Auto-Complete Timeout

**REPLACES**: "Auto-complete after configurable timeout" from original plan.

If the orchestrator delivered a draft and is waiting for feedback, it waits indefinitely. The coroutine is cheap (awaiting an `asyncio.Event`). No CPU, no memory pressure.

If user wants a new task while one awaits feedback: "Task #3 is waiting for your feedback. Want me to auto-approve it and start your new task?" Natural LLM conversation, not code logic. Or user explicitly parks the task (#32).

### Decision #36: task_context Table (Replaces Session Notes via heartbeat_events)

**REPLACES**: Session notes piggybacking on `heartbeat_events` table.

New table specifically for task lifecycle context that nanobot needs:

```sql
CREATE TABLE IF NOT EXISTS task_context (
    id TEXT PRIMARY KEY,
    task_id TEXT NOT NULL,
    created_at TEXT NOT NULL,
    context_type TEXT NOT NULL,  -- 'completion', 'question', 'progress', 'failure'
    summary TEXT NOT NULL,
    detail TEXT,
    acknowledged INTEGER DEFAULT 0,
    acknowledged_turns INTEGER DEFAULT 0
);
```

Key differences from heartbeat_events:
1. **Not acknowledged on first read.** `acknowledged_turns` increments each time nanobot sees it. After N turns (default 5), moves to acknowledged.
2. **Survives compaction.** Read from SQLite each turn, not from message history.
3. **Queryable by task.** User asks "what happened with task #3?" → nanobot queries `task_context WHERE task_id = '3'` regardless of session state.

Heartbeat events remain a separate mechanism for their original purpose (ephemeral system observations).

### Decision #37: Heartbeat Model Upgrade — 20B Local + Cloud Fallback

**REPLACES**: 4B SLM heartbeat from original plan.

- **Primary**: 20B local model (LM Studio) when LM Studio is running.
- **Cloud fallback** (when LM Studio is down — expected, not abnormal):
  1. Latest Gemini Flash free tier (Google AI Studio endpoint) — $0.00. Default heartbeat cloud model. Weekly review verifies and updates model version as Google releases new versions.
  2. Llama 4 Scout (OpenRouter free tier) — $0.00
  3. Gemini Flash paid (OpenRouter) — ~$0.003/tick
- **Daily cost**: $0.00 when using free tiers. ~$0.04/day if all free tiers exhausted. ~$15/year worst case.
- **Rationale**: 4B model cannot do "conversation reflection" or "opportunity spotting." 20B model can. The cost is negligible.
- **Keep 4B SLM for**: Embeddings, real-time extraction, classification — speed-critical tasks where reasoning isn't needed.

### Decision #38: models.md Scope — Orchestrator/Worker Selection Only

`models.md` remains focused on worker model selection by the orchestrator. It is NOT used for:
- Heartbeat model selection (infrastructure config in `CopilotConfig`)
- Conversation model selection (infrastructure config in `CopilotConfig`)
- Orchestrator model selection (infrastructure config in `CopilotConfig`)

All infrastructure model assignments live in `CopilotConfig` with dedicated fields. `models.md` is the orchestrator's "menu" for assigning models to worker agents within a task workflow.

### Decision #39: Model Selection Review → Weekly Cycle Exclusively

**REPLACES**: Any model review responsibility in the dream cycle.

The model landscape changes too fast for daily review without current data. Model selection review belongs exclusively to the weekly review cycle (Tier 4), where:
- Web search is MANDATORY for the reviewing LLM
- The review examines: new model releases, pricing changes, free tier changes, benchmark results
- Cross-references with actual task performance data from the week
- Proposes changes to `models.md` AND infrastructure config model assignments
- All proposals go through the dream observation staging area (user approval required)

The dream cycle (daily) can FLAG model performance concerns ("Gemini Flash failed 4/10 research tasks") but does NOT recommend alternatives. It writes observations for the weekly review to act on.

### Decision #40: Free API Priority + Cost Awareness

**Orchestrator guidance** (add to `orchestrator_soul.md`):
```
When selecting models for workers, prefer models with free tiers for
routine tasks (research, extraction, formatting):
- Gemini 3 Flash (Google AI free tier)
- Llama 4 Scout (OpenRouter free tier)
- GLM-5 (Z.AI 20M free tokens)
Reserve paid models for tasks requiring specific capabilities
(strong reasoning, code generation, long context).
```

**Free tier tracking** — new `free_tier_status` table:
```sql
CREATE TABLE IF NOT EXISTS free_tier_status (
    model TEXT NOT NULL,
    provider TEXT NOT NULL,
    status TEXT DEFAULT 'active',
    -- active: free tier available
    -- daily_limit_reached: hit daily quota, resets at daily_reset_at
    -- credit_exhausted: permanent — free credits fully consumed (e.g. NIM 5K credits)
    -- unknown: never checked or ambiguous billing response
    last_checked TEXT,
    daily_calls_used INTEGER DEFAULT 0,
    daily_limit INTEGER,           -- provider's daily cap (NULL if unknown/unlimited)
    daily_reset_at TEXT,           -- ISO timestamp when daily quota resets
    credit_remaining INTEGER,      -- for credit-based tiers (NULL if daily-based)
    PRIMARY KEY (model, provider)
);
```

- After each API call, check response cost. If `response_cost = 0` → free tier active. If billed → check why:
  - If `daily_calls_used >= daily_limit` → set `daily_limit_reached`. Heartbeat/scheduler checks `daily_reset_at` and flips back to `active` after reset.
  - If credit-based and `credit_remaining = 0` → set `credit_exhausted`. This is permanent — model is removed from free pool until weekly review re-evaluates.
- Before dispatching a worker, orchestrator checks: "Is this model's free tier `active`?" Skip `daily_limit_reached` and `credit_exhausted` models.
- If a supposedly-free API starts billing unexpectedly, fire a `medium` AlertBus alert.
- Daily reset logic: lightweight check in heartbeat tick — if `status = 'daily_limit_reached' AND now > daily_reset_at`, reset `status = 'active'` and `daily_calls_used = 0`.

### Decision #41: WhatsApp Outbound Rate Limiter

Simple guardrail — not complex infrastructure:
- Max 1 message per 2 seconds to the same `chat_id`
- FIFO queue for excess messages
- If queue exceeds 10 messages, merge oldest into a digest
- ~20 lines in the WhatsApp channel handler
- No crashes, no deadlocks, no approval-system-style problems

### Decision #42: Identity File Pointer Retrieval

When identity files have `<!-- memory: keyword1, keyword2 -->` pointers to evicted content:

**At orchestrator startup**:
1. Load identity files
2. Extract `<!-- memory: ... -->` tags
3. Run Qdrant vector search for extracted keywords
4. Append retrieved context as `## Deep Context` section below identity file content

**In ExtendedContextBuilder** (for nanobot conversation):
- Same pattern: detect pointer tags, run Qdrant search, append
- Reuses existing proactive episodic memory mechanism
- Within existing 2-second timeout

Cost: ~0ms (local embedding) + ~50ms (Qdrant). No LLM calls.

### Decision #45: Situational Awareness Briefing — Automatic Context Injection

**PROBLEM**: When a new session starts, or the model switches, the LLM has no knowledge of active tasks, pending questions, recent completions, or session history. The key insight: important state is in structured stores (SQLite, task_context, session metadata) — but the model won't spontaneously call tools to check these on every message. It doesn't even know what to call.

**SOLUTION**: `ExtendedContextBuilder` injects a **Situational Awareness Briefing** into the system prompt on every message. This is a lightweight SQLite-queried block — no LLM calls, no tool invocations by the model. The context builder does the work before the model ever sees the message.

**Injection format** (appended to system prompt by ExtendedContextBuilder):

```
## Current Situation

### Active Tasks
- Task #abc: "Research VPS providers" — active, 2/3 steps complete, worker running
- Task #def: "Draft weekly report" — parked (user requested)

### Pending Questions
- Task #abc needs your input: "Should I include providers outside the US?"
  (asked 2h ago, unacknowledged)

### Recent Completions (last 24h)
- Task #ghi: "Compare project management tools" — completed 2h ago.
  Summary: Compared 5 tools, recommended Linear. Full report in task history.

### Session Context
- Current model: Haiku 4.5 (default conversation)
- Model switched from Sonnet 4.5 → Haiku 4.5 (user: "/use haiku") 3 messages ago
- Today's spend: $1.23 (conversation $0.45, tasks $0.62, background $0.16)
```

**Data sources** (all SQLite queries, no LLM):
| Section | Query |
|---------|-------|
| Active Tasks | `SELECT * FROM tasks WHERE status IN ('active', 'parked', 'awaiting_user') ORDER BY priority` |
| Pending Questions | `SELECT * FROM task_context WHERE context_type = 'question' AND acknowledged = 0` |
| Recent Completions | `SELECT * FROM task_context WHERE context_type = 'completion' AND created_at > datetime('now', '-24 hours')` |
| Session Context | Session metadata (`last_model_used`, switch history) + `routing_log` cost aggregation |

**Characteristics**:
- **Always injected** — not opt-in, not tool-based. The model sees this before processing the user message.
- **Cheap**: 4 SQLite queries, ~100-300 tokens of output. No LLM calls.
- **Survives compaction**: Rebuilt from SQLite every turn, not dependent on message history.
- **Survives model switches**: New model immediately knows full situation.
- **Survives new sessions**: `/new` clears messages but SQLite state persists. New session gets full briefing.
- **Gracefully empty**: If no tasks/questions/completions, section is omitted entirely. Zero overhead for simple conversations.
- **ExtendedContextBuilder owns it**: New method `_build_situational_briefing()` alongside existing `_build_identity_context()`, `_build_heartbeat_context()`, etc.

**What this replaces**: The hope that the model will "know" to check tools. The hope that context bridging notes are sufficient. The hope that the user will re-state context. None of those work reliably. Direct injection does.

### Decision #46: File & Document Handling

**PROBLEM**: WhatsApp file handling has four bugs: (1) media files deleted before agent reads them, (2) non-image documents silently ignored, (3) files >25MB silently dropped, (4) captionless documents/images/video dropped entirely. Beyond bugs, the V2.1 architecture has no design for how files reach orchestrators/workers.

**Bugs fixed** (implemented in this session):

1. **Persistent media storage**: WhatsApp bridge saves to `~/.nanobot/media/` (matching Telegram/Discord), not `/tmp/`. No immediate deletion. Document filenames include message ID to prevent collision.
2. **Auto-extraction**: `ContextBuilder._build_user_content()` now extracts text from PDFs (PyMuPDF), Excel (openpyxl), and text files inline — appended to the user message. The LLM sees document content automatically. Guarded imports for optional deps; graceful fallback on failure.
3. **Rejection as information**: Files >25MB included in bridge payload as `rejectedFiles[]`. Python channel appends `[File rejected: name (32.1MB) — exceeds 25MB limit]` to message text. LLM responds naturally. No blocking, no approval system.
4. **Captionless media**: `extractMessageContent()` now returns `[Document: filename]`, `[Image]`, `[Video]` for media without captions. Previously returned null, dropping the entire message.

**V2.1 design** (deferred implementation):

5. **task_attachments table**: Files attached to tasks persist with SQLite reference. Workers access attachments by task ID.
```sql
CREATE TABLE IF NOT EXISTS task_attachments (
    id TEXT PRIMARY KEY,
    task_id TEXT NOT NULL,
    filename TEXT NOT NULL,
    file_path TEXT NOT NULL,
    mime_type TEXT,
    size_bytes INTEGER,
    created_at TEXT NOT NULL,
    FOREIGN KEY (task_id) REFERENCES tasks(id)
);
```

6. **Orchestrator delegates document work**: The orchestrator NEVER processes document content directly. For documents requiring analysis:
   - Dispatch a document ingestion worker (cheap model — Gemini Flash, GLM-5)
   - Worker extracts, chunks, and summarizes
   - Orchestrator receives summary + chunk index
   - If deeper analysis needed, orchestrator dispatches workers to process specific chunks
   - This applies to book editing: orchestrator sees chapter outline, dispatches per-chapter editing workers

7. **Model capability checks**: Before sending an image to a worker, verify the model supports vision (litellm model metadata or explicit capability map in `models.md`). If not, auto-extract text description via a vision-capable model, or route to one.

8. **Large file strategy**: Files >25MB cannot come through WhatsApp. User provides a URL or local file path in the message. Orchestrator dispatches a worker to fetch/process. No binary pipeline needed.

9. **Web upload endpoint** (V2.2+): For book-sized content (100K+ words), a simple web interface for direct file upload to `~/.nanobot/media/`. Also enables CLI file support. WhatsApp remains for quick documents (<25MB). The web endpoint writes the file and creates a `task_attachments` row; nanobot picks it up on next interaction.

10. **Media cleanup** (deferred): Heartbeat or cron deletes files in `~/.nanobot/media/` older than 7 days, unless referenced by `task_attachments`. Not building this now — WhatsApp compresses aggressively, directory grows slowly.

## Open Questions Remaining

1. **Default conversation model**: Haiku 4.5 ($1/$5) vs Gemini Flash ($0.50/$3.00)? Haiku is better quality, Gemini Flash is cheaper and has a free tier. User preference needed.
2. **Gemini Flash free tier limits**: Weekly review is responsible for verifying and documenting current limits (Decision #39). Heartbeat defaults to latest Gemini Flash free tier (Decision #37). Limits will be tracked dynamically via `free_tier_status` table.
3. **`acknowledged_turns` threshold for task_context**: Default 5 turns proposed. Should this vary by context_type? (completion notes probably need more turns than progress updates)
4. **Memory consolidation target model**: Should use at least a mid-tier model. Which specific model? Should be configurable in CopilotConfig.
5. **Provider priority per tier**: When multiple cloud providers are configured (OpenRouter, Venice, NVIDIA), the failover chain tries them in config insertion order with no explicit priority. The new `ConversationProvider` should support per-tier provider priority — e.g., prefer NVIDIA for cheap calls, OpenRouter for frontier models.
