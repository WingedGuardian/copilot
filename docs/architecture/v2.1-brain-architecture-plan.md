### Failure Analysis Mitigations (2026-02-17)

Forward-looking failure analysis ("what could go wrong halfway through?") produced 4 additional decisions:

1. **Identity file 1000-token soft cap** — Weekly review (Tier 4) consolidates when exceeded. Evicted content moves to long-term memory (vector store). Identity files retain pointer keywords (`<!-- memory: keyword1, keyword2 -->`) so higher-tier models search memory for depth. SLM reads the compact version.
2. **Dream observation quality gate** — User must approve all observations. Nothing auto-applies. Weekly review resurfaces relevant pending observations. First 30 days: fully manual (bootstrap period). After 30 days: revisit. Each observation tracks `supporting_event_count`.
3. **Data retention** — Store forever. ~30MB/year for all tables combined. Not a real problem. More data = smarter learning loop.
4. **AFK message batching** — User-configurable via `/settings notifications [immediate|digest|priority]`. Default: digest. Queue non-urgent messages; digest on return.

### Architecture Review (2026-02-17, later session)

Critical review of the full design identified over-architecture, under-architecture, contradictions, and anticipated bugs. Key changes:

- **Fold dispatcher into orchestrator**: `WorkerDispatcher` was ~15 lines of indirection. Dispatch logic lives in `orchestrator.py`. One module owns everything.
- **Inline eval → soul file**: Mandatory code-level eval removed from execution loop. `orchestrator_soul.md` guides when to evaluate. LLM decides, not code.
- **Iteration loop added**: Orchestrator delivers DRAFT → blocks for feedback → re-plans if needed → delivers again. Only terminates on user approval or configurable timeout. Closes the ITERATE step in the task lifecycle.

- **Cancellation mechanism added**: `_running_orchestrators` dict on TaskWorker + cancellation events + asyncio task cancel. Natural language cancel through nanobot.
- **Delivery UX**: All direct with self-identification prefix (`[Task #3 — Orchestrator]`). System outbound never loops back as inbound.
- **Bootstrap workflow confirmation**: First 30 days, orchestrator shows plan before executing.
- **SqlitePool mandate**: All DB access through pool. No more raw `aiosqlite.connect()`.
- **Phase 0 added**: ~10 infrastructure tasks testable with current codebase before brain architecture.
- **Bug fixes**: `get_next_pending()` → pending only, crash recovery for stuck `awaiting` tasks, tool name→class registry.

### Critical Architectural Principles

**1. Interface Agent and Orchestrator are SEPARATE SESSIONS.**
The Interface Agent is nanobot's existing agent loop — always available, conversational model, the "ears and mouth." When it detects a task, it creates a record and hands it off. It does NOT execute the task. It stays free for conversation 24/7.

**2. The Orchestrator is a fresh, ephemeral high-power agent spawned per task.**
Not the same session as the conversation. Not persistent across tasks. A NEW agent (frontier model) spins up, designs the workflow, manages workers to completion, then terminates. Avoids context contamination between tasks.

**3. Workers are ephemeral, specialized SubAgents spawned by the orchestrator.**
Each gets a specific model, tools, system prompt, and input. They run via the existing SubagentManager. Multiple workers can run in parallel via asyncio.gather. The orchestrator crafts their identity and specialization — no hardcoded agent types.

**4. The orchestrator delivers directly to the user — and defaults to silence.**
The user does NOT want progress updates. A task should be silent unless: (a) the orchestrator needs something from the user (clarifying question, approval for a consequential action), (b) the task is complete with the deliverable, or (c) the task has failed in a way the orchestrator can't resolve. The user can proactively check status via `/tasks` or `/task <id>`. The orchestrator delivers messages directly to the user via the message bus — NOT through nanobot's LLM (no telephone game, no token waste, no lost-in-translation risk). Nanobot receives a session note (structured context update, no LLM processing) so it has context for follow-up conversations. Workers never communicate with the user directly. Same pattern applies to dream cycle and weekly review deliveries.

**5. The orchestrator diagnoses failures — it doesn't brute-force retry.**
When a worker fails, the orchestrator receives the error and partial output, then reasons about WHY it failed before deciding what to do. Was it a transient error? A model capability gap? Missing information that needs to be surfaced to the user? A fundamentally wrong approach that needs re-planning? The orchestrator is frontier-tier intelligence — let it think about the problem, don't prescribe a decision tree in code.

**6. No blanket policies unless there's a specific reason.**
The orchestrator is an intelligent agent. Prescribing rigid rules for it (like "always retry with bigger model" or "always review worker output") is the same mistake as the regex task detector. Put principles and context in `orchestrator_soul.md`; let the orchestrator make contextual decisions. The only hard constraints are cost budget limits and permission gates for consequential actions.

## Agent Identity & Organizational Awareness

Every agent in the system — nanobot, orchestrator, workers — must understand the organization it operates within. Not just WHAT to do, but WHERE it fits, WHO it talks to, and WHY the system is designed this way. An agent that doesn't understand its role in the larger picture will make decisions that work locally but fail systemically.

### Nanobot (Interface Agent) — Must Know the Full Picture
Nanobot is the persistent conversational agent. Its context files (`capabilities.md`, `agents.md`) must explain:

**What the task system is and why it exists**: "You are the interface — the ears and mouth — of a larger brain. When the user asks you to DO substantial work (research, build, compare, analyze), you don't do it yourself. You create a task, acknowledge it, and hand it off to an orchestrator that designs and manages the work. You stay free for conversation. The user is the CEO; you're the receptionist and the intercom, not the worker."

**When to create tasks vs. answer directly**: "If the user asks a question (What is X? How does Y work?), answer it. If they ask you to DO something that requires multiple steps, real research, or will take more than a quick response — create a task. The dividing line: if you'd need tools, time, or multiple steps to produce a quality deliverable, it's a task."

**What happens after a task is created**: "An orchestrator — a fresh, frontier-tier agent — spins up for this task specifically. It reads soul files, designs a workflow of parallel and sequential steps, spawns specialized workers, evaluates their output, handles failures, and delivers the result directly to the user. You don't manage this. You receive session notes about what was delivered so you can handle follow-up conversations."

**How to handle task-related interactions**: "You'll see pending task questions in your context. When the user replies with something that answers a task question, use the TaskTool's resume action. When they ask about task status, use /tasks or /task. When a task completes, the orchestrator delivers directly to the user — you receive a session note with context so you can handle follow-up questions."

**WHY this design**: "Because doing real work takes time, multiple models, and shouldn't block conversation. The user wants to text a task from their phone and get a result hours later. They don't want to babysit. They want you available to chat about other things while work happens in the background."

### Orchestrator — Must Know Its Role and Relationships
Every time an orchestrator spawns, it loads its soul files + the model pool + capabilities + best practices. These must explain:

**`data/copilot/orchestrator_soul.md`** — The orchestrator's complete identity:
- **Who you are**: "You are the chief of staff — a fresh, per-task agent with frontier-level intelligence. You exist for ONE task. You design the workflow, pick the right models, craft the right prompts, manage workers, handle failures, and deliver the result. When you're done, you terminate."
- **Who nanobot is**: "Nanobot is the interface agent — the user's conversational companion. It created this task and handed it to you. You deliver messages directly to the user via the message bus. Nanobot gets a session note (structured context update) so it can handle follow-up conversations, but your messages go straight to the user — no telephone game."
- **When to communicate**: "Silent by default. Only contact the user when: you need input (clarifying question, approval), the task is complete (deliverable), or you've hit an unrecoverable failure. The user doesn't want progress updates. They'll check /tasks if they're curious."
- **Self-review**: "Before delivering your final result, review your own workflow: Did the workers produce what was specified? Did you evaluate their output? Is the deliverable actually what the user asked for? Review your work before presenting it."
- **WHY silence**: "Because the user is the CEO. They delegate and check in on their schedule. Noise erodes trust. If you're doing your job well, they shouldn't hear from you until you have something to show."
- **How to design workflows**: Design principles, DAG structure, model assignment from the pool, when to parallelize, how to handle dependencies
- **How to handle failures**: "Diagnose before acting. Why did it fail? Transient error, model too weak, missing info, wrong approach? The answer determines the action. Don't brute-force retry."
- **Inline evaluation**: "Before accepting a worker's output, evaluate it. Is this actually good enough? Does it answer what was asked? Is the confidence appropriate? A bad result that propagates through the DAG wastes everything downstream."
- **WHY you exist as a separate agent**: "Because context contamination between tasks degrades quality. Because frontier-level intelligence is needed for decomposition — it's the highest-leverage decision. Because the interface agent shouldn't be blocked by task work."
- **Learning awareness**: "You are part of a learning organism. You are ephemeral, but the system is not. Every decision you make — which models you chose, how you structured the workflow, how you diagnosed failures, what you'd do differently — persists beyond your lifespan via the audit trail and your post-task retrospective. A nightly review cycle examines your decisions alongside their outcomes and extracts patterns that update the very soul files and best practices that future orchestrators will load. Log your reasoning alongside your decisions. Write an honest retrospective before you terminate. Every failure is an opportunity for insight — not just for this task, but for every task that follows."

**`data/copilot/models.md`** — The model pool menu with capabilities, costs, free tiers.

**`data/copilot/capabilities.md`** — Available tools, what the system can and cannot do.

**`data/copilot/agents.md`** — Worker best practices learned over time.

### Worker Principles (`data/copilot/worker_principles.md`)
Universal principles injected into EVERY worker's context, regardless of specialization:

**Who you are**: "You are an ephemeral specialist — spawned for one specific job within a larger workflow. You were created by an orchestrator that designed this workflow, picked your model, gave you specific tools, and crafted your instructions."

**Who you report to**: "You report to the orchestrator, NOT to the user. If you're stuck, uncertain, or need to flag something, include it clearly in your output. The orchestrator reads your output and decides what to do next."

**Core principles**:
- Ask rather than assume — if the task is ambiguous, say so in your output rather than guessing
- Don't take consequential actions (spending money, sending messages, modifying external systems) without flagging them
- Cite sources, show your work, be transparent about confidence
- If stuck, return what you have with an honest assessment — don't hallucinate completion
- Structure your output: result, confidence level, any warnings or caveats
- Re-read the task spec before delivering output — does your result actually match what was asked?

**WHY these principles**: "Because your output feeds into other workers' inputs. A wrong assumption here cascades. Because the orchestrator evaluates your work before accepting it — honesty about uncertainty helps it make better decisions. Because the user's trust in the system depends on every component being trustworthy."

**Learning awareness**: "You are part of a learning organism. Your output — including your confidence levels, caveats, and honest assessments of uncertainty — persists beyond your lifespan. It feeds into a nightly review cycle that extracts patterns and improves future task execution. Every honest uncertainty flag, every cited source, every caveat you include makes the SYSTEM smarter over time. Every hallucinated confidence or uncited claim makes it dumber. You won't see the benefit — you're ephemeral. But the next worker spawned for a similar job will be better because of what you reported honestly."

### Worker Identity (crafted per-job by orchestrator)
On top of the universal principles, the orchestrator crafts a complete worker context per job:
- **System prompt**: Specialized for THIS job ("You are a research specialist. Your job is to find and synthesize information about X. Be thorough, cite sources, present findings in structured format.")
- **Model selection**: Based on job requirements (cheap/fast for simple research, strong for synthesis, local for privacy)
- **Tool set**: Only the tools this worker needs
- **Input data**: Results from prior workers, formatted as context
- **Expected output**: What "done" looks like for this worker

Dynamic specialization, not fixed agent types. The orchestrator's intelligence (and its soul files) determine the quality of specialization.

### Persistent Best Practices (`data/copilot/agents.md`)
As the system runs tasks, patterns emerge:
- "Research workers perform better with Gemini Flash than Haiku for web search tasks"
- "Writing workers need at least Sonnet-class for quality output"
- "Always include 'cite your sources' in research worker prompts"

These get captured (manually at first, dream cycle later) and loaded into the orchestrator context every time. This is how the system gets better over time without retraining.

## Learning Loop — How the System Gets Smarter

The task execution system must feed into the same self-learning infrastructure that V1's conversational loop uses. Ephemeral agents (orchestrator, workers) don't learn individually — the SYSTEM learns through persistent storage + dream cycle review.

### What gets captured (during task execution)

**Automatically (no agent awareness needed):**
- Cost per node (provider tracks it)
- Duration per node (dispatcher timestamps)
- Success/failure status (workflow state machine)
- User feedback on final deliverable (goes through normal extraction pipeline)

**Requires agent awareness (would be MISSED otherwise):**
- **Orchestrator reasoning** — Not just WHAT it decided (the workflow JSON), but WHY. "Chose Gemini Flash for research because cheap + good at web search" vs. just `"model": "gemini-flash"`. Stored alongside the workflow design in SQLite. Without the reasoning, the dream cycle sees outcomes but can't learn WHY the orchestrator's choices worked or failed.
- **Worker confidence and uncertainty** — Workers must include structured metadata (confidence level, caveats, source reliability assessments) in their output. This isn't bureaucracy — it's training data. A worker that says "pricing data for provider C came from a 2024 source, confidence: medium" gives the dream cycle richer signal than one that presents a table without qualification.
- **Failure diagnosis context** — When the orchestrator diagnoses a worker failure, that reasoning must be persisted in the workflow audit trail, not just used to decide the next action and discarded. "Worker failed because Haiku couldn't handle 50K-token research context — switched to Sonnet and it worked" is exactly what should update `agents.md`.
- **Post-task retrospective** — Before terminating, the orchestrator writes a brief "what worked, what didn't, what I'd do differently" to SQLite using its own model (Tier 3/4). Without this, the orchestrator's accumulated judgment about the task dies when it terminates — the dream cycle never sees it.
- **Re-planning events** — When and why the orchestrator changed the workflow mid-execution, including the diagnosis that led to the change.

### What reviews it (dream cycle)
The nightly dream cycle adds a new job: **task execution review**. It examines completed tasks from the past 24 hours and extracts patterns:
- Which model assignments worked well? Which didn't? → Updates `agents.md` best practices
- Which orchestrator prompting patterns produced good worker output? → Updates `orchestrator_soul.md` refinements
- Were there recurring failure modes? → Updates `capabilities.md` with limitations discovered
- Did the user correct or reject deliverables? → Feeds into the existing lesson system with confidence/decay

### What improves (identity files)
The learning loop closes through the identity files that get loaded every time an orchestrator spawns:
- **`agents.md`** — Worker best practices, model performance observations, prompting patterns
- **`orchestrator_soul.md`** — Decomposition strategies that work, failure handling improvements
- **`models.md`** — Real-world cost and quality observations per model (supplements the static specs)
- **`worker_principles.md`** — New universal principles discovered from mistakes

The dream cycle writes proposed updates to a `dream_observations` SQLite table with reasoning — NOT directly to identity files. Proposed changes are reviewable via `/dream` command. **User must approve all observations** — nothing auto-applies. Weekly review (Tier 4) resurfaces relevant pending observations. Each observation tracks `supporting_event_count` so user can see the evidence basis. No manual review of raw logs; the system does the analysis and presents actionable proposals in a staging area.

### Bootstrap Period (First 30 Days)
During the first 30 days of operation, ALL learning is user-supervised:
- All dream observations require manual `/dream` approval
- All identity file changes require explicit user sign-off
- No auto-application of lessons to identity files
- Weekly review resurfaces outstanding approvals that are still relevant

This protects against wrong conclusions from small sample sizes. The system needs enough data (task outcomes, user feedback, cost patterns) before its observations carry statistical weight. After 30 days, the weekly review can propose relaxing the gate based on observed quality of dream cycle proposals.

### Learning Hierarchy
```
Weekly Review (Tier 4) → improves → Dream Cycle + Heartbeat references
Dream Cycle (Tier 2/3) → improves → Heartbeat reference + reviews heartbeat quality
Heartbeat (Tier 0)     → reads reference → writes observations → NEVER judges higher tiers
```
Bigger models judge smaller. Never the reverse. SLM can be a "fresh set of eyes" but must never override frontier-tier decisions.

### Recursive Self-Review
The system reviews its own work at every tier. Each tier reviews the tier below it. Dream cycle and weekly review ALSO evaluate system-level outcomes (user satisfaction, delivered value), not just component mechanics.

```

  Catches: Surface-level misses, obvious gaps
  Scope: Component-level (conversations, tasks, infrastructure)

Tier 2/3 — Orchestrator Inline Eval (per-worker)
  Reviews: Worker output
  Against: Expected output spec, task requirements
  Catches: Bad results before they cascade through the DAG

Tier 3/4 — Orchestrator Retrospective (per-task, orchestrator model)
  Reviews: Its own workflow design + model assignments
  Against: Actual outcomes, worker performance
  Catches: Wrong decompositions, model misassignments
  Note: Written by the orchestrator itself before terminating — uses
    the orchestrator model (user-designated, typically frontier/Tier 4)

Tier 2/3 — Dream Cycle (daily)
  Reviews: Orchestrator decisions + heartbeat events + blackboard
  Against: Task outcomes, user feedback, cost efficiency
  Catches: Recurring patterns, failure modes, identity file improvements
  ALSO reviews: Overall system value delivery — are tasks completing well?
    Is the user correcting results? Are deliverables useful? Is the system
    getting BETTER over time or stagnating?
  ALSO asks: What skills do we need to BUILD to make our lives easier?

internal mechanics but actual value delivered to the user.
```

This principle is encoded in identity files: `orchestrator_soul.md` ("review your own workflow before delivering") and `worker_principles.md` ("re-read the task spec before delivering output").

### Connection to existing V1 learning
- **Lesson system**: Task outcomes create lessons the same way conversational feedback does. "User rejected the comparison table because it lacked pricing details" → lesson with confidence decay.
- **Qdrant episodic memory**: Task context and deliverables are stored for semantic recall. Next time a similar task comes in, the orchestrator can retrieve how it was handled before.
- **Extraction pipeline**: User feedback on task deliverables goes through the same background extraction that conversational messages use.

## Coordination Between Agents

### How results flow (the DAG)
The workflow is a directed acyclic graph (DAG), not a flat list:
- Each node has a `depends_on` list (which nodes must complete first)
- Independent nodes run in parallel
- When node A completes, its result is injected into node B's input prompt. The orchestrator applies output budgets or summarization if the result exceeds the downstream node's context budget — full text only when sizes are manageable.
- All results stored in SQLite in full for persistence and debugging (summarization is for piping, not storage)

### Shared workspace (for tasks that produce files)
For V2.2+ when workers can write files:
- Each task gets a workspace directory: `/tmp/nanobot/tasks/{task_id}/`
- Workers read/write files there
- The orchestrator can reference workspace paths in worker prompts
- Final deliverables are collected from the workspace

### SQLite as coordination backbone
SQLite is the right choice for a single-machine system:
- Workflow state (which nodes are running/complete/failed/blocked)
- Full result text per node (no filesystem coordination needed for V2.1)
- Cost tracking per node
- Audit trail of all orchestrator decisions

This is simpler and more reliable than filesystem coordination for our use case. We're not on a Mac Mini with 6 agents reading files — we're on one server with async coroutines and a database.

## How the Orchestrator Works (the key new piece)

The orchestrator is a per-task async coroutine (NOT a SubAgent itself — it needs to spawn SubAgents). When the TaskWorker picks up a pending task, it launches an orchestrator coroutine:

1. **Load context**: Read orchestrator SOUL files, model pool, capabilities, best practices
2. **Design workflow**: Direct `provider.chat()` call (frontier model) with the task description + context → output a workflow DAG as structured JSON. The prompt asks for reasoning alongside decisions — not just `"model": "gemini-flash"` but WHY that model for that node. Reasoning stored in `design_reasoning` (workflow-level) and `reasoning` (per-node).
3. **Bootstrap confirmation** (first 30 days): Show workflow design to user: `[Task #X — Orchestrator] I plan to do this in N steps: [summary]. Budget ~$X. Proceed?` Block until approved. After bootstrap period, skip this step (silent by default).
4. **Persist workflow**: Store nodes + edges + reasoning in SQLite (via SqlitePool)
5. **Check cancellation**: Before each dispatch, check `cancellation_event.is_set()`. If set, abort gracefully.
6. **Execute ready nodes**: Spawn workers (SubAgents via `SubagentManager.spawn()`) for all nodes with no unmet dependencies. Workers dispatched using LiteLLM native routing — model name resolves to provider via env vars. Run in parallel via asyncio.gather. Dispatch logic lives in `orchestrator.py` (no separate dispatcher).
7. **Pipe results**: When a node completes, store its result. Check if any blocked nodes are now ready. If so, inject results and dispatch them.
8. **Handle failures**: When a worker fails, the orchestrator receives the error message and any partial output. It makes a direct `provider.chat()` call to diagnose the failure — not just "retry or abort" but: Was this a transient error (retry same worker)? A capability gap in the model (retry with different model or approach)? Missing information (surface a question to the user)? A wrong strategy (re-plan the remaining workflow)? An unrecoverable issue (report failure with explanation)? The orchestrator reasons about the failure with full context before acting.
9. **Handle questions**: If the orchestrator determines it needs user input — whether from initial analysis, mid-workflow discovery, or failure diagnosis — it blocks the task (set pending_questions) and delivers question directly: `[Task #X — Orchestrator] I need to know: {question}`. Blocks until user answers (nanobot routes reply via TaskTool resume → asyncio.Event wakes orchestrator).
10. **Deliver draft + iterate**: When all nodes are done, assemble the deliverable and deliver as a DRAFT **directly to the user** via message bus: `[Task #X — Orchestrator] Here's the draft: {deliverable}. Does this meet your needs?` Block for feedback. If user approves → proceed to completion. If user gives feedback → re-plan (add/modify nodes), re-execute, deliver updated draft. Auto-complete after configurable timeout if no response. Send session note to nanobot with structured context (task ID, summary, status).
11. **Post-task reflection**: Before terminating, the orchestrator writes a brief retrospective to SQLite: what worked, what didn't, what it would do differently, which model assignments exceeded or fell short of expectations. This uses the orchestrator's own model via direct `provider.chat()` (user-designated, typically Tier 3/4) — the same intelligence that designed and managed the workflow is the one reflecting on it. This is the richest learning material for the dream cycle — without it, the orchestrator's reasoning dies when it terminates.

## Key Design Decisions

### 1. LLM-based task detection, not regex
The LLM IS the intelligence. Principle 9 from V2 Architecture: "Let the LLM decide." The microscopic latency/cost is more than acceptable. The system should be a tool of the LLM, not subsidizing its work.

### 2. Workflows are DAGs, not flat lists
Steps have dependencies. Independent steps run in parallel. Results flow between steps. This is the minimum viable structure for real work.

### 3. Workers are ephemeral and dynamic, not fixed agent types
No hardcoded "Researcher" or "Writer" classes. The orchestrator designs each worker's configuration (model, tools, prompt) on the fly. Specialization comes from the orchestrator's intelligence and its SOUL files, not from code.

### 4. Per-worker model assignment
The orchestrator picks the right model for each job. Cheap for simple, frontier for complex, local for privacy. The model pool (`data/copilot/models.md`) is the orchestrator's menu.

### 5. Per-task cost budget with runtime enforcement
Before executing each worker, check cumulative cost. Abort if over budget. Default $0.50, configurable per task.

### 6. Result chaining with output budgets
Worker B's input includes Worker A's output. Full results stored in SQLite. However, the orchestrator applies output budgets or summarization for large results before piping to downstream nodes — "no truncation" was naive and would blow context windows for cheap models. Each WorkflowNode has an optional `output_budget` field; the orchestrator summarizes oversized output before injection.

### 7. Identity files loaded every time
The orchestrator gets its SOUL files + model pool + capabilities + best practices every time it's spawned. Workers get specialized prompts crafted by the orchestrator. This is how quality compounds over time.

## Research-Informed Design Additions

Based on web research (DeepMind/MIT scaling study, OpenAI Self-Evolving Agents, Columbia DAPLab failure analysis, ACM CSCW trust study):

**11. Inline evaluation after each worker.** The orchestrator evaluates worker output before accepting it — not just pass/fail but quality assessment. Don't wait for the dream cycle to catch bad output that propagates through dependent nodes. This is a cheap LLM call (coordinator-tier) that prevents expensive cascading errors.

**12. Structured results, not strings.** Worker output is structured: `{result, confidence, warnings, model_used, duration_ms}`. Feeds observability, evaluation, and the learning loop. A string `result` field can't distinguish "completed successfully" from "completed but uncertain" from "failed but produced something."

**13. Concurrent worker cap of 3-4 per task.** DeepMind found coordination tax eats gains beyond ~4 agents. The orchestrator should design workflows with this constraint. Config-driven via `max_parallel_workers`.

**14. Circuit breakers for transient failures.** Exponential backoff with jitter on API failures. If N consecutive workers fail with the same error class (rate limit, timeout), stop dispatching and wait. Prevents retry storms.

**15. Task detection via existing TaskTool — no new module.** The LLM already has `TaskTool` registered. The agent already decides when to call tools. We just need clearer identity file guidance about when to create tasks. No `detector.py`, no separate classification call. This is Principle 9 fully realized.

**16. Feedback routing via existing agent intelligence.** Same principle — when tasks have pending questions, inject that context into the agent's prompt. The LLM sees "Task #abc is waiting for your answer about budget" and naturally routes the user's reply. No `feedback.py` needed.

## Resolved Design Decisions
1. **Orchestrator is per-task, not persistent.** Fresh agent spawned per task. Avoids context contamination. The Interface Agent (nanobot) is the persistent one.
2. **V2.1 tools: research only.** Web search + web fetch. Proves the architecture without tool complexity.
3. **LLM-based everything.** Task detection, feedback routing, workflow design — the LLM decides.
4. **Dynamic specialization, not fixed agent types.** The orchestrator crafts each worker's identity on the fly, informed by persistent best-practice files.
5. **SQLite coordination.** Right choice for single-machine system. Simpler and more reliable than filesystem coordination.
6. **Recurring workflows are a future enhancement.** Save workflow templates for repetitive tasks (V2.x), but the core architecture is dynamic-first.
7. **Silent by default.** User hears about a task at creation (ack), when input is needed, and at completion. No progress updates. User checks status proactively via `/tasks` or `/task <id>`.
8. **Orchestrator delivers directly to the user.** Workers never talk to the user. The orchestrator decides what's worth surfacing and delivers directly via the message bus — NOT through nanobot's LLM. Nanobot receives session notes (structured context updates) for follow-up awareness. Same pattern for dream cycle and weekly review deliveries.
9. **Failure = diagnosis first, action second.** The orchestrator reasons about WHY a worker failed (transient error? model too weak? missing info? wrong approach?) before deciding what to do. No blanket retry policies.
10. **Principles over policies.** The orchestrator is an intelligent agent — put principles and context in `orchestrator_soul.md`, not hardcoded decision trees in code. The only hard constraints are cost budgets and permission gates for consequential actions (spending money, sending messages, modifying external systems).

### Task 2.2: Orchestrator Coroutine [L]
**Create**: `nanobot/copilot/brain/orchestrator.py`
**Depends on**: 0.1, 0.3, 0.5, 1.1, 2.1

The core new component — async coroutine `run_orchestrator(task, provider, subagent_manager, task_manager, db_path, notify_fn, config, docs_dir)` (dispatch logic inline, no separate dispatcher):

1. **Load context** — read soul files from docs_dir
2. **Check for existing workflow** — if nodes exist in DB, resume (crash recovery)
3. **Design workflow** — frontier model LLM call → parse JSON → persist to SQLite
4. **Handle clarifying questions** — if returned, set pending_questions, notify, return
5. **Execution loop**:
   a. Get ready nodes (max `config.max_parallel_workers`)
   b. **Check cancellation** — `cancellation_event.is_set()` before each dispatch
   c. Check budget before each dispatch (atomic SQL — see baked-in fixes)
   d. **Dispatch workers** — inline in orchestrator (no separate dispatcher). Prepend `worker_principles.md`, call `SubagentManager.spawn()` with model/tools/prompt/callback. Workers dispatched via LiteLLM native routing (model name → provider via env vars). Run in parallel via `asyncio.gather`.
   e. For failures: direct `provider.chat()` diagnosis call → decide action (retry/remodel/replan/ask user/abort)
   f. Circuit breaker: 3 consecutive same-error failures → stop dispatching, notify
   g. Pipe completed results to downstream nodes
6. **Deliver draft + iterate** — assemble deliverable, deliver as DRAFT with self-identification prefix: `[Task #X — Orchestrator] Here's the draft: {deliverable}. Does this meet your needs?` Block for feedback via pending_questions. If approved → continue. If feedback → re-plan, re-execute, deliver updated draft. Auto-complete on configurable timeout.
7. **Post-task reflection** — direct `provider.chat()` call (orchestrator's own model, Tier 3/4) writes retrospective with 4 structured questions. Stored in workflow's `retrospective` field.
8. **Complete** — mark task done, send session note to nanobot, terminate.

**Note on inline evaluation**: NOT a mandatory code step. `orchestrator_soul.md` includes: "Evaluate worker output before accepting, especially when downstream nodes depend on it." The orchestrator decides when quality matters. It can also design explicit verification nodes into the DAG. See Decision #17.

**Baked-in structural fixes (from failure analysis — build these in from day 1, not as patches):**
- **DAG cycle detection [Gap 3.3]**: `from_orchestrator_json()` runs topological sort on the parsed DAG. Cycles → reject with error message to orchestrator LLM for re-design.
- **JSON retry with error feedback [Gap 3.2]**: If orchestrator's workflow design JSON fails to parse, retry up to 3 times with the parse error injected as context. Not silent retries — the LLM sees what went wrong.
- **Max retry count per node [Gap 7.2]**: Default 3 retries per node, regardless of error diversity. Prevents infinite retry loops when the orchestrator keeps trying different approaches on a fundamentally broken node.
- **Model validation [Gap 3.4]**: Before dispatching a worker, validate the assigned model exists in provider config. Fail fast with a clear error rather than a cryptic provider exception.
- **In-memory result fallback [Gap 6.3]**: If DB write fails after a worker completes, hold the result in memory and continue the pipeline. Log the DB failure to `heartbeat_events`. Don't lose completed work because of a transient SQLite issue.
- **Downstream-aware summarization [Gap 6.1]**: When summarizing oversized node output for downstream injection, the summarization prompt includes the downstream node's `expected_output` field. The orchestrator designed the DAG — it knows what matters for each consumer.

**CRITICAL**: Orchestrator makes direct `provider.chat()` / `litellm.acompletion()` calls for its own reasoning (Decision #22). Does NOT use `process_direct()`. Workers use SubagentManager which creates separate asyncio tasks. LiteLLM native routing handles multi-provider dispatch via env vars (Decision #23).

**Verify**:
- Integration test with mock provider: 2-node DAG → mock workers → results chain → task complete
- Clarifying questions path → task goes to `awaiting`
- Budget enforcement → task fails with explanation
- Worker failure → diagnosis → retry with different model
- Crash recovery → load existing workflow, resume from last completed node

## LLM vs Code — Gap Categorization

Not every gap needs code. The LLM is always the pilot. Judgment calls belong in soul file guidance; structural concerns belong in code.

| Gap | Disposition | Why |
|-----|------------|-----|
| 1.1/1.2 Task misclassification | **LLM** (agents.md guidance) | The LLM IS the classifier — better guidance, not code guardrails |
| 3.5 Bad workflow design | **LLM** (orchestrator_soul.md: "review your own workflow before dispatching") | Frontier model self-review catches design flaws |
| 5.4 Garbage passing eval | **LLM** (better eval prompt in orchestrator_soul.md) | Code can't judge research quality — only the LLM can |
| 6.1 Summarization losing info | **LLM** (orchestrator prompt: "summarize preserving what downstream node needs") + **Code** (downstream `expected_output` injection) | The orchestrator designed the DAG, it knows what matters. Code helps by providing the downstream context. |
| 7.1 Diagnosis failure fallback | **LLM** (soul file: "if you can't diagnose, pause and tell the user") | A principle, not a decision tree |
| 8.3 Ambiguous reply routing | **LLM** (natural disambiguation with injected context) | LLM handles "$20" → "budget?" trivially when pending questions are in context |
| All others (1.3, 1.5, 2.1, 3.2, 3.3, 3.4, 4.1, 5.1, 5.3, 6.3, 7.2, 8.1, 8.4, 9.3, 9.4, 12.3) | **Code** (structural) | Timeouts, validation, event wiring, atomic SQL, concurrency fixes — not judgment calls |

**Principle**: This is the same insight that killed the approval system. Don't build code that overrides LLM judgment. Build code that gives the LLM better information and handles the mechanical plumbing.

## Task Pipeline Monitoring Architecture

How task pipeline events flow through **existing** infrastructure — no new monitoring systems.

```
TASK PIPELINE MONITORING (wires into existing infrastructure)
─────────────────────────────────────────────────────────────
Detection (code, zero LLM cost):
  asyncio.wait_for timeout → blackboard + AlertBus
  Budget 80% threshold    → blackboard + AlertBus (medium)
  Budget 100% exceeded    → AlertBus (high) + abort
  Circuit breaker trip    → AlertBus (medium) + blackboard
  Worker no activity >5m  → cancel + AlertBus (medium)
  Rate limit 429          → backoff tracker + AlertBus (low)

Auto-resolution (code, zero LLM cost):
  Hung API call → cancel + retry with backoff (existing pattern from channels/manager.py)
  Rate limit   → exponential backoff with jitter (existing pattern)
  Model unavail → next provider in failover chain (if RouterProvider wired)
  Budget 80%   → warn user, continue. 100% → abort gracefully.

Escalation (when auto-resolution fails):
  AlertBus high severity → deliver_fn → OutboundMessage → WhatsApp
  Nanobot sees heartbeat_events at next turn → conversational delivery
  User sees: "Task #abc hit a rate limit. I tried backoff + model switch.
  Still failing. Task paused — want me to retry or cancel?"

KEY: Goes THROUGH nanobot's conversation, not raw bus injection.
The old approval system failed by bypassing nanobot. This doesn't.
```
