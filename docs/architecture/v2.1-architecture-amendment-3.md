## Decisions

### Decision #36: Task completion requires explicit user approval — no auto-complete timeout

**Amends**: Decision #21 (design review notes), which said "No response for configurable timeout → auto-complete."
**Superseded by**: Amendment #2 Decision #35 (indefinite wait) — this decision confirms and makes permanent.

Tasks are not complete until the user explicitly approves. There is no timeout that auto-approves a delivered draft. Tasks that are delivered but not approved stay in "delivered" status indefinitely. If the user wants to move on, they can park the task (`/park`) or explicitly approve/reject it.

**Rationale**: The user is the product owner. Auto-approval risks delivering subpar work and eroding trust — especially in the early days when the system hasn't proven itself.

---

### Decision #37: Mandatory plan approval before execution — no 30-day sunset

**Amends**: Decision #26 (design review notes), which said "After 30 days, goes silent per normal policy."

The orchestrator ALWAYS shows the workflow plan and cost estimate before executing. There is no automatic sunset. The user will explicitly tell nanobot when they're ready to relax this requirement (e.g., `/settings tasks auto-approve-plans`).

**Rationale**: User wants manual control over when to extend trust. A timer-based sunset doesn't account for whether the system has actually earned that trust.

---

### Decision #38: Sequential task queue in V2.1 — parallel execution deferred to V2.2

**Amends**: Brain architecture plan and V2 Architecture doc, which described parallel orchestrators via `asyncio.create_task()`.

V2.1 executes one task at a time. Tasks queue by priority. User can park a running task to start a different one (`/park <id>`, `/resume <id>`). Parallel execution moves back to V2.2.

**Rationale**: Simpler, more predictable, lower complexity. The task threshold (only substantial multi-step work becomes a task) means trivial requests never enter the queue — so queue blocking is a non-issue.

---

### Decision #39: Two-phase intake interview before task creation

**New addition** — not in any prior design doc.

Task creation follows a two-phase intake process:

**Phase 1 — Nanobot-side intake** (latency-sensitive):
1. LLM detects task potential from user message
2. Confirms with user: "This sounds like a task. Is that right?"
3. Gathers requirements, desired outcomes, success criteria
4. Creates task record, spawns orchestrator

**Phase 2 — Orchestrator-side intake** (thoroughness-optimized):
1. Orchestrator reviews everything nanobot gathered
2. Can immediately come back with MORE questions before starting any work
3. Only proceeds to plan design after it's satisfied with the information

**Key insight**: The two-phase intake is structurally necessary, not redundant. Nanobot is latency-sensitive (user in live conversation, can't wait 1+ minutes). The orchestrator has zero latency pressure — thoroughness and accuracy are what matter. Different time constraints produce different quality of analysis even from identical models. The orchestrator WILL catch things nanobot misses because it's not racing to respond.

**No heuristics**. LLM makes the classification call. No explicit `/task` trigger required (though the user accepts this ambiguity and expects self-learning to improve classification over time).

### Decision #40: Task threshold — deliverable + iteration

**New addition** — clarifies what constitutes a "task" vs "chat."

A message becomes a task when:
1. The user expects a **deliverable** (not just an answer) — a file, repo, deployed artifact, structured document
2. Completing it will likely require **multiple iteration cycles** of back-and-forth

Examples that ARE tasks: "Build me a landing page," "Set up a GitHub repo and scaffold a Python project," "Build out this entire RFP from these documents."

Examples that are NOT tasks: "Research the top 3 VPS providers" (nanobot can do this), "Cancel my 2pm meeting" (MCP/skill call), "Draft me a quick email" (nanobot with tool).

The bright line: **would you deliver something above and beyond answering a question?**

### Decision #41: Indefinite blocking on user questions — never auto-proceed

When the orchestrator needs user input on an ambiguous decision, it waits indefinitely. The system never auto-proceeds with "best judgment" on decisions where the user's authority matters.

**Rationale**: Respects user authority. Assumptions lead to wasted work and eroded trust. The user's response time is their prerogative.

---

### Decision #42: Worker tools for V2.1 — research + files only, no shell

V2.1 workers get:
- Web search + web fetch (research)
- File read/write (create documents, edit files)

V2.1 workers do NOT get:
- Shell / exec (deferred to V2.2 with coding agent)
- Direct WhatsApp send (workers communicate through orchestrator only)

**Rationale**: Security concern. Shell access requires the capability profile system (Decision #44) to be safe. Workers in V2.1 are research + writing workers, not coding workers.

---

### Decision #43: Concurrency UX — start verbose, configurable to silent

When the user texts nanobot while an orchestrator is executing, nanobot:
- Prioritizes task-related messages (feedback, cancels)
- Proactively mentions running task status when contextually relevant
- Still handles normal chat

This is the "verbose" mode. The user can switch to "silent" mode (orchestrator runs fully in background, chat is fully independent) via `/settings tasks verbose|silent` or by telling nanobot directly.

**Rationale**: Verbose until trust is established. User graduates to silent when they're confident tasks complete reliably.

### Decision #44: Capability profiles replace flat sandbox

**Amends**: The design's tools_whitelist concept. Extends it to a full capability profile.

Different task types need different security models:

| Task Type | Internet | Filesystem | AWS | Shell | Browser |
|-----------|----------|-----------|-----|-------|---------|
| Research | Allowlisted domains | Read outside, write inside task dir | None | None | None |
| Code/Repo | GitHub API only | Task workspace + git | None | Sandboxed | None |
| AWS Deploy | AWS APIs only | Task workspace | Scoped IAM role per task | AWS CLI only | None |
| Web Form-Fill | Target site only | Task workspace | None | None | Playwright isolated profile |
| Content/Writing | Web search | Task workspace | None | None | None |

The orchestrator assigns a capability profile to each workflow node when designing the DAG. The worker spawner enforces it. This extends `tools_whitelist` to include: tools + network scope + filesystem scope + credentials.

Key security properties:
- AWS: Orchestrator creates scoped IAM session with only needed permissions. Never exposes root credentials.
- Browser: Fresh Playwright profile per task. No access to user's sessions/cookies.
- Credentials injected per-service by orchestrator, logged in audit trail.

### Decision #45: Simplified crash recovery — one retry then halt

**Amends**: Brain architecture plan's crash recovery design.

Simplified model:
1. State persisted to SQLite at every DAG node checkpoint
2. On restart: auto-spawn orchestrator from last checkpoint
3. Orchestrator runs health check
4. **First crash**: Auto-resume with brief notification to user
5. **Second crash on same task**: Always halt regardless of cause, notify user
6. Crash counter per task, resets on pause/complete/cancel
7. Nanobot helps diagnose the issue with the user

**Removed**: Causal analysis ("was the task the cause?"). Too unreliable in practice — LLM will frequently guess wrong or say "inconclusive." Simple retry count is more predictable.

### Decision #46: Cost management — running meter, not upfront estimate

**Amends**: Brain architecture plan's cost tracking.

V2.1 does NOT attempt upfront cost estimation (too inaccurate without historical data). Instead:

1. User pre-selects orchestrator model
2. Orchestrator tracks actual spend as it executes (per-node cost tracking already in design)
3. When cumulative cost crosses $5 threshold, system pauses: "This task has cost $X so far with Y steps remaining. Continue?"
4. User approves or aborts

**Self-learning path**: Structured `cost_analysis` block written on every task completion:
```json
{
  "estimated_cost": null,
  "actual_cost": 4.80,
  "cost_by_step_type": {"research": 1.20, "writing": 2.40, "review": 1.20},
  "model_usage": {"opus": 3.10, "sonnet": 1.70},
  "iterations": 2,
  "task_type": "rfp_document"
}
```

Dream cycle observes patterns → proposes estimation improvements → eventually the orchestrator CAN do upfront estimates based on historical data for similar task types.

**Targets for self-learning** (seeded, refined by data):
- Cost estimation accuracy: within 30% of actual → improve to 15%
- Steps estimated vs actual: within 1 → exact
- Avg iterations to approval: track trend, aim to decrease

### Decision #47: Graceful failure handling — workers finish, then pause

When a worker hits an unresolvable error:
1. Other workers finish their current step (with a 5-minute timeout on the "graceful finish" phase — after that, workers are killed)
2. Blocked worker escalates to orchestrator
3. Orchestrator pauses the task
4. Orchestrator asks user through nanobot (via pending_questions)
5. User responds naturally
6. Orchestrator evaluates if the answer resolves the blocker
7. If yes → brief notification + auto-resume
8. If no → follow-up questions

---

### Decision #48: Cancellation with confirmation + force-close

When user says "cancel task 3":
1. Nanobot confirms: "Cancel task 3? X steps complete, Y in progress. Confirm?"
2. User confirms → graceful stop (workers finish current step, partial results saved)
3. If user sends a SECOND cancel request → hard stop (workers killed immediately)

Two-tier: first request = graceful with confirmation. Second request = force-close.

### Decision #49: Learning asymmetry fix — playbook system for successes

**New addition.**

Successes get structured capture, not just narrative retrospectives:

**Playbook system**: Reusable workflow templates synthesized from successful task completions.
- Stored in SQLite `task_playbooks` table + Qdrant for semantic search
- Schema includes: task_type, workflow_template (nodes + model recommendations), user_preferences, lessons, avg cost, success_rate
- Lifecycle: First task → no playbook. After 2-3 successful similar tasks → dream cycle proposes playbook → user approves. Future tasks use playbook as starting template.
- Orchestrator queries playbooks during workflow design. `/playbooks` command for user to view/edit.

**Learning balance**:
- Successes → memory system + playbooks (known-good approaches, build repertoire)
- Failures → proposed changes to identity files, prompts, workflows, tools (evolve the system)
- Neither auto-applies to identity files without user approval

---

### Decision #50: Feedback routing — only applies to active task

User feedback via WhatsApp only ever applies to the currently executing task. Queued tasks cannot receive feedback — only instructions for nanobot to note and pass to the orchestrator when appropriate.

Example: User texts "for that next task B with the website, make sure we use Tailwind." Nanobot notes this and passes it to task B's orchestrator when task B starts — not while task A is running.

### Decision #51: Web research mitigations — 10 additions to the design

Based on research across Devin AI, AutoGPT, CrewAI, multi-agent failure studies (Galileo AI, 41-87% failure rates across 1,642 traces), LangGraph production deployments, and WhatsApp bot architectures:

| # | Mitigation | Where | Why |
|---|-----------|-------|-----|
| 1 | **Blocker detection with early bailout** | `orchestrator_soul.md` + retry counter per node | Devin spent days on impossible tasks. "If 3 approaches failed, STOP and ask the user." |
| 2 | **Task feasibility check** | First-class orchestrator step before DAG design | Orchestrator assesses "can I do this?" before decomposing |
| 3 | **WhatsApp webhook deduplication** | `bridge/whatsapp/` message processing | WhatsApp retries until 200 → duplicate task creation without dedup |
| 4 | **Prompt caching** | Provider config + orchestrator context | `cache_control` on soul/policy/capabilities prefixes. Up to 90% savings. |
| 5 | **Context windowing** | Orchestrator context builder | Re-inject original task + DAG state at each decision. LLMs forget by step 15. |
| 6 | **Schema-validated worker outputs** | Worker result handler | Parse failure → retry with correction. Prevents cascading garbage. |
| 7 | **Progress heartbeats to WhatsApp** | Node completion handler | Tasks > 30s get acknowledgment. Updates at each node. Critical for async UX trust. |
| 8 | **Idempotent worker operations** | `worker_principles.md` + idempotency keys | Re-execution doesn't duplicate side effects. |
| 9 | **Start narrow: 3-5 task types in V2.1** | Scope constraint | 95% of GenAI pilots fail when general-purpose (MIT 2025). V2.1: research docs, code scaffolding, file deliverables. |
| 10 | **Prompt injection defense** | Orchestrator context injection | User input from WhatsApp isolated in delimited blocks. Never concatenated into system prompt. |
