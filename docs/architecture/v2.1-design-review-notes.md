## Decisions Made

### 1. Orchestrator/Dream Cycle/Weekly Review → User: DIRECT delivery
No telephone game through nanobot's LLM. These agents format their own messages for the user and deliver directly via the message bus. Nanobot gets a session note (structured context update, no LLM processing) so it maintains context for follow-up conversations.

| From → To | Path | LLM involved? |
|---|---|---|
| Orchestrator → User | Direct via bus | No |
| Dream cycle → User | Direct via bus | No |
| Weekly review → User | Direct via bus | No |
| Heartbeat → Nanobot | Blackboard (user never sees) | SLM only |
| Nanobot ← User reply | Normal conversation | Yes |

### 2. Routing tiers: Use the 5-tier model from V2 Architecture
| Tier | Name | Runtime | Key role |
|---|---|---|---|
| 0 | Brainstem SLM | Local (4B) | Heartbeat, extraction |
| 1 | Local Cortex | Local (20-30B) | Conversation, privacy |
| 2 | Tactical Hub | Cloud (fast) | Metacognition, dream cycle |
| 3 | Cognitive Core | Cloud (strong) | Task creation, complex reasoning |
| 4 | Executive Office | Cloud (frontier) | Weekly review, architecture |

### 3. Model assignments for background services
- **Orchestrator**: Hard-set config parameter (not routed). User controls via `/use orchestrator <model>`. Shown in `/status`. Typically Tier 3/4 (frontier). The orchestrator's own retrospective uses this same model — the intelligence that designed the workflow is the one reflecting on it.
- **Dream cycle**: Hard-set Tier 2 minimum, Tier 3 for analytical jobs. Reviews the ENTIRE agentic workflow output — not just the orchestrator's self-review, but worker outputs, heartbeat events, and system-level patterns.
- **Weekly review**: Hard-set Tier 4.
- **Heartbeat**: Hard-set Tier 0. Never cloud.

### 4. TaskTool only available at Tier 3+ *(amended — see Decision #21)*
~~Structurally enforced — tool removed from definitions for Tier 0/1/2.~~ **Revised**: tier restriction via provider type check. RouterProvider's `classify()` step determines whether the resolved provider is local (< Tier 3) or cloud (≥ Tier 3). If local → exclude TaskTool from tool definitions. If cloud → include. This works because LM Studio connection = local models = Tier 0-2; no LM Studio = cloud only = Tier 3+. The router already knows this — we just need to split `classify()` from `chat()` to expose the decision before tool assembly. Self-escalation (Lesson #3) handles edge cases: identity files for lower tiers say "If you recognize the user wants substantial work done but you cannot create tasks, escalate."

### 5. SubagentManager worker default: SILENT
Workers spawned by the orchestrator must default to silent (no announce). Add `silent=True` flag to `spawn()` that suppresses `_announce_result()`. Prevents raw worker output leaking to user via bus fallback.

### 6. Blackboard is queryable, not injected
Nanobot does NOT get the full blackboard on every message. Heartbeat writes to SQLite. Nanobot queries it on demand via `/dashboard` (programmatic — no LLM, pure SQLite query formatted as text, same pattern as `/tasks`). Only HIGH-priority items get pushed as lightweight context notes (~20-50 tokens). Dream cycle reviews full 24h of blackboard.

### 7. Dream cycle staging area (user-approved, not auto-apply)
Dream cycle writes proposed changes to `dream_observations` SQLite table with reasoning — NOT directly to identity files. Reviewable via `/dream` command. **User must approve all observations** — nothing auto-applies. Weekly review (Tier 4) resurfaces pending observations that are still relevant. First 30 days: fully manual approval (bootstrap period — prevents wrong conclusions from small samples). After 30 days: revisit whether to relax the gate. Each observation tracks `supporting_event_count` so user can see the evidence basis.

### 8. Heartbeat feeds dream cycle
Dream cycle Job 11 reads `heartbeat_events` alongside workflow audit trail — full picture of what orchestrator planned, what happened, and what went wrong at infrastructure level. Heartbeat feeds dream cycle, doesn't duplicate it.

## Critique Points (from review)

### Critical (experience killers)
1. **Bus announce fallback** — SubagentManager's `_announce_result()` leaks raw worker output to user if `result_callback` isn't set. Fix: silent default for orchestrator workers.
2. **60-second poll + sequential tasks** — TaskWorker polls every 60s and processes one task at a time. Fix: `asyncio.Event` wake signal + launch orchestrators as independent `asyncio.create_task()` coroutines so multiple tasks run in parallel.

### Important (will cause problems)
3. **"No truncation" result chaining** — Full worker output piped to next node blows context windows. Fix: orchestrator applies output budgets or summarization for large results. Add `output_budget` field to WorkflowNode.
4. **Crash recovery lacks orchestrator context** — New orchestrator has no memory of original strategy. Fix: persist a recovery briefing (workflow state + completed results + design reasoning) that a replacement orchestrator reads. Not just "reset running nodes to pending."
5. **Orchestrator API keys** — Dedicated LiteLLMProvider bypasses RouterProvider's key management and failover. Fix: ensure orchestrator has access to provider registry API keys.
6. **SQLite write contention** — Multiple concurrent writers. Fix: `PRAGMA journal_mode=WAL` in schema migration.
7. **Worker tools not restricted** — SubagentManager gives all tools. Fix: `tools_whitelist` CREATES only listed tools (not all-minus-blacklist). Security boundary: workers should NOT have tools the orchestrator didn't assign.
8. **Orchestrator cost invisibility** — Orchestrator's own LLM calls (workflow design, evaluation, diagnosis, retrospective) tracked in `cost_log` so they're visible to dream cycle cost analysis.

### Design tensions
9. **Lesson #2 vs Design Decision #1** — Heuristic routing for conversation (proven), LLM-based for task detection (untested). Not a contradiction given different contexts, but local model task detection needs structural guard (TaskTool restriction).
10. **Learning loop quality** — Dream cycle auto-updating identity files risks wrong conclusions. Fix: staging area with approval, not auto-apply (see Decision #7).

## The Heartbeat: Reframed as Proactive Peripheral Vision

**NOT a health check.** It's the system's ambient awareness — the mechanism that makes the system proactive. Nanobot is reactive NOW; the goal is proactive. The heartbeat + blackboard is HOW nanobot becomes proactive. Nanobot with a blackboard IS a proactive agent. The heartbeat is the mechanism that makes it so.

**Role split**: Heartbeat = immune system (real-time detection), dream cycle = learning system (reflection + improvement). Same domains, different depth/model.

### Design principles
- Runs on Tier 0 SLM (4B). NEVER cloud.
- NEVER takes action. Only writes observations.
- NEVER talks to the user. Writes to blackboard. Nanobot decides what to surface.
- NEVER judges higher-tier work. Observes, doesn't second-guess. But acts as COURIER — orchestrator retrospectives pass through as-is without filtering.
- Most ticks should be SILENT (nothing significant to report).

### Living reference: `data/copilot/heartbeat.md`
- 200-300 tokens max (4B SLM constraint)
- Defines: what to check, significance thresholds, quality standards, current focus areas
- Improved by: dream cycle (nightly), weekly review (weekly), manual edits, nanobot convos

### Baseline checks (every tick, programmatic, zero LLM cost)
- Task status: any stuck/completed/failed tasks?
- Orchestrator liveness (deferred to pre-2.2/V2.2 — see below)
- Cost burn rate anomalies
- Undelivered messages

### Rotating roster (SLM analysis focus areas cycle across ticks)
| Tick | Focus |
|---|---|
| 1 | Conversation reflection — dropped threads? |
| 2 | Temporal awareness — deadlines, follow-ups? |
| 3 | Anomaly detection — cost patterns, errors? |
| 4 | Opportunity spotting — recurring themes, connections? |

Note: These are the SAME domains the dream cycle reviews — but at different depth. Heartbeat catches surface-level, obvious stuff (SLM). Dream cycle does deep analysis with full context (Tier 2/3).

### Push threshold
The bar for pushing to nanobot is HIGH — there must be *some* reasoning behind the observation, not just pattern-matching. "User mentioned landing page twice" alone = don't push (too thin). "User mentioned landing page twice, there's a task template from a similar past request, and the user has free time tonight" = definite push (actual context and reasoning behind it).

### Learning hierarchy
```
Weekly Review (Tier 4) → improves → Dream Cycle + Heartbeat references
Dream Cycle (Tier 2/3) → improves → Heartbeat reference + reviews heartbeat quality
Heartbeat (Tier 0)     → reads reference → writes observations → NEVER judges higher tiers
```

Bigger models judge smaller. Never the reverse. SLM can be a "fresh set of eyes" but must never override frontier-tier decisions.

## Recursive Self-Review — Architectural Principle

The gap analysis exercise itself (reviewing our work against conversation source material) exemplifies a core system capability: recursive self-review at every tier.

```
RECURSIVE SELF-REVIEW HIERARCHY
────────────────────────────────────────────────────────────────────
Tier 0 — Heartbeat (every 2h, SLM)
  Reviews: Recent conversations, task statuses, system state
  Against: Dropped threads, unanswered questions, commitments without tasks
  Catches: Surface-level misses, obvious gaps
  Scope: Component-level (conversations, tasks, infrastructure)

Tier 2/3 — Orchestrator Inline Eval (per-worker)
  Reviews: Worker output
  Against: Expected output spec, task requirements
  Catches: Bad results before they cascade through the DAG

Tier 3/4 — Orchestrator Retrospective (per-task, orchestrator model)
  Reviews: Its own workflow design + model assignments
  Against: Actual outcomes, worker performance
  Catches: Wrong decompositions, model misassignments
  Note: Written by the orchestrator itself before terminating — uses
    the orchestrator model (user-designated, typically frontier/Tier 4)

Tier 2/3 — Dream Cycle (daily)
  Reviews: Orchestrator decisions + heartbeat events + blackboard
  Against: Task outcomes, user feedback, cost efficiency
  Catches: Recurring patterns, failure modes, identity file improvements
  ALSO reviews: Overall system value delivery — are tasks completing well?
    Is the user correcting results? Are deliverables useful? Is the system
    getting BETTER over time or stagnating?
  ALSO asks: What skills do we need to BUILD to make our lives easier?
    Nanobot is skill-driven (homemade skills, not externally downloaded).
    Draw inspiration from what's out there, but build in-house when
    feasible and cost-effective.

Tier 4 — Weekly Review (weekly)
  Reviews: Dream cycle quality + heartbeat quality + system trajectory
  Against: User goals, stated priorities, satisfaction signals
  Catches: Whether the review mechanisms themselves are working
  ALSO reviews: The ENTIRE nanobot+copilot system as a product —
    delivered value, user satisfaction, missed opportunities, strategic
    direction. "Are we building the right thing? Is the user's life
    materially better this week than last?"
  ALSO asks: What does NANOBOT need to BECOME? Not just capabilities
    and features, but the system and the thinking within it. What might
    need to be rearchitected? What should change, be added, or removed
    to deliver the most value?
────────────────────────────────────────────────────────────────────
Principle: Each tier reviews the tier below it. Dream cycle and weekly
review ALSO review the system's outcomes holistically — not just
internal mechanics but actual value delivered to the user.
```

This pattern manifests at every level: heartbeat (dropped threads), orchestrator inline eval (worker output), orchestrator retrospective (own decisions), dream cycle (system-wide patterns), weekly review (quality audit of the review mechanisms themselves).

## Context Continuity Across Compaction/Restart

Blackboard persists in SQLite — survives everything. New context briefing includes:
1. User profile (USER.md, ~50 tokens)
2. Active task summaries (1-line per task from SQLite)
3. Recent HIGH-priority blackboard items (max 3-5, `attention`/`urgent` only)
4. Pending task questions awaiting user response

Compaction should avoid firing when:
- Tasks have pending questions (question context would be lost)
- User is mid-conversation about a specific task

Heartbeat can help: programmatic check for "is context window getting large?" and pre-write a concise briefing to SQLite for the compaction process.

## Failure Analysis Decisions (2026-02-17)

Following a forward-looking failure analysis ("what could go wrong halfway through, not just day 1?"), four architectural decisions were made:

### 9. Identity file token budget: 1000-token soft cap
Identity files (orchestrator_soul.md, worker_principles.md, heartbeat.md, etc.) have a 1000-token soft cap. When exceeded, weekly review (Tier 4) consolidates — rewrites the file for coherence and moves evicted content to long-term memory (vector store). Identity files retain pointer keywords (`<!-- memory: keyword1, keyword2 -->`) so higher-tier models know to search long-term memory for depth. The SLM reads the compact version; Tier 2+ follow breadcrumbs when they need more context.

This means identity files become a curated index, not the full knowledge store. The system's accumulated knowledge grows without bound in long-term memory while the working set stays SLM-friendly.

### 10. Dream observation quality gate: user-approved, weekly review surfaces
Nothing auto-applies. Weekly review resurfaces relevant pending observations for user approval. First 30 days: fully manual (bootstrap period). After 30 days: revisit whether to relax the gate. See Decision #7 (amended above).

### 11. Data retention: store forever (~30MB/year)
SQLite data retention was flagged as a failure mode but the math doesn't support it: heartbeat events (~2MB/yr), blackboard (~2MB/yr), dream observations (~7MB/yr), cost log (~15MB/yr), workflow audit (~2.5MB/yr) = ~30MB/year total. Not a real problem for a single-user system. More data makes the learning loop smarter — dream cycle analyzing 6 months of cost trends is better than 30 days. Just add timestamp indexes (standard practice).

### 12. AFK message batching: user-configurable notifications
`/settings notifications [immediate|digest|priority]`. Default: digest. Queue non-urgent messages during AFK periods; deliver a single digest when user sends first message after gap >2h. HIGH priority (task failures, cost alerts) delivers immediately regardless of setting.

## Task System Failure Analysis (2026-02-17)

End-to-end failure analysis traced a task through 12 lifecycle stages (intake → wake → design → dispatch → execution → result chaining → failure handling → user interaction → delivery → completion → learning → concurrent tasks). Identified 20 failure modes. All critical/high claims verified against actual code.

### 6 Critical gaps (user-visible failures day 1-2)
1. **TaskTool tier restriction not implemented** (1.3) — `ToolRegistry.get_definitions()` returns all tools unconditionally. No tier filtering anywhere.
2. **session_key not passed by TaskTool** (1.5) — `execute()` never receives session context. Tasks created without session affinity.
3. **No timeout on SubagentManager LLM calls** (5.1) — bare `await provider.chat()` with no `asyncio.wait_for`. `cancel_stuck` only runs every 2 hours.
4. **Pending questions never injected into agent context** (8.1) — zero references to pending questions in context builder. LLM has no idea tasks are waiting.
5. **No wake mechanism after resume** (8.4) — `clear_pending_questions()` updates DB only. No event/signal to orchestrator.
6. **Session note mechanism not designed** (9.3) — mentioned 6+ times in plan, never specified. No implementation or stub exists.

### 6 High gaps (task failures under normal operation)
7. **No DAG cycle detection** (3.3) — one bad LLM JSON output creates permanently stuck task
8. **No retry for malformed workflow JSON** (3.2) — even frontier models occasionally produce invalid JSON
9. **No global worker limit** (4.1) — 10 tasks × 3 workers = 30 API calls = rate limit thundering herd
10. **process_direct concurrency bug** (5.3) — mutates `self.model` across await boundaries, 4+ concurrent callers
11. **Cost budget TOCTOU race** (12.3) — two orchestrators check budget simultaneously, both pass, both dispatch
12. **No max retry count per node** (7.2) — circuit breaker keyed on "same error class," diverse errors bypass it

### 8 Medium gaps (degraded quality or edge cases)
3.4 no model validation, 6.1 summarization uninformed by downstream, 4.2 tools_whitelist unimplemented, 4.3 silent=True error path leaks, 2.1 stuck-task threshold fights orchestrators, 6.3 no in-memory result fallback, 9.4 delivery-then-crash wrong state, 7.1 diagnosis failure no fallback

### 13. LLM-first design principle
Prefer LLM judgment (soul file guidance, identity file updates, better prompts) over programmatic guardrails for decision-making. Reserve code for structural concerns (timeouts, data validation, event wiring, atomic SQL). The LLM is always the pilot in nanobot.

**LLM-handled gaps** (soul file guidance, not code):
- 1.1/1.2 Task misclassification → `agents.md` guidance (the LLM IS the classifier)
- 3.5 Bad workflow design → `orchestrator_soul.md`: "review your own workflow before dispatching"
- 5.4 Garbage passing eval → better eval prompt in orchestrator_soul.md
- 6.1 Summarization losing info → orchestrator prompt: "summarize preserving what downstream node needs"
- 7.1 Diagnosis failure → soul file: "if you can't diagnose, pause and tell the user"
- 8.3 Ambiguous reply routing → LLM naturally disambiguates with injected context

### 14. Monitoring through existing infrastructure
No new monitoring system. Wire task pipeline events into existing AlertBus (`copilot/alerting/bus.py`), heartbeat_events table, ProcessSupervisor, and deliver_fn pattern. Detection → auto-resolution → escalation THROUGH nanobot's conversation (not raw bus injection). Session notes reuse the heartbeat_events table pattern — no new table needed.

### 15. Six-phase implementation *(amended — added Phase 0)*
Phase 0: Infrastructure Wiring (pre-brain plumbing fixes) → Phase 1: Foundation (data models, schema, identity files) → Phase 2: Core Engine (orchestrator + critical structural fixes baked in) → Phase 3: Wiring (integration gaps, session notes, question injection) → Phase 4: Resilience + Monitoring (global semaphore, rate limits, AlertBus wiring, auto-resolution) → Phase 5: Polish + Learning (memory, identity consolidation, blackboard reframe). System is fully robust before use.

## Architecture Review Decisions (2026-02-17, later session)

Critical review of the full V2.1 design identified over-architecture, under-architecture, design contradictions, anticipated bugs, and hard problems. 17 decisions made.

### 16. Fold dispatcher into orchestrator
The `WorkerDispatcher` is ~15 lines of logic translating a `WorkflowNode` into a `spawn()` call. Not worth a separate file, class, or phase dependency. Fold into `orchestrator.py`. One module owns dispatch. Phase 4's semaphore and rate-limit tracker become private state of the orchestrator module.

### 17. Inline evaluation → soul file guidance
Move mandatory inline eval (code, step 5d of execution loop) to `orchestrator_soul.md` as a principle: "Evaluate worker output before accepting, especially when downstream nodes depend on it." The orchestrator decides when quality matters — for V2.1 (simple research), mandatory eval triples cost for little benefit. For V2.2+ (coding), the orchestrator will add verification nodes to the DAG itself. **Tradeoff**: if a worker produces garbage and the orchestrator doesn't evaluate, it cascades. Mitigated by soul file guidance + the orchestrator can design explicit verification nodes when risk is high.

### 18. Notification preferences: keep but defer
The 3-mode notification system (immediate/digest/priority) stays in the plan but deferred to Phase 5 or later. "Silent by default" + existing alert mute is sufficient for current message volume (~5-10/day). Build when notification fatigue is actually a problem.

### 19. "Delivered" status: keep (belt + suspenders)
Keep the `delivered` status alongside Task 4.9's activity-based stuck detection. Two complementary mechanisms: explicit state transition + heuristic activity check.

### 20. Delivery UX: all direct with self-identification prefix
All orchestrator/dream cycle/weekly review deliveries go direct to WhatsApp with mandatory prefix identifying the source:
- `[Task #3 — Orchestrator] Here are the VPS comparison results...`
- `[Dream Cycle — Daily Report] Today's observations...`
- `[Weekly Review] Week in review...`

**Inbound routing**: All user messages come through nanobot (single inbound channel). Nanobot sees session notes in context and routes task-related replies via TaskTool. System outbound messages never loop back as inbound — completely separate code paths (inbound = WhatsApp webhook with user's phone, outbound = `bus.publish_outbound()`). No confusion between user messages and system messages.

### 21. Iteration loop: draft + feedback + re-plan
The orchestrator delivers a DRAFT, not a final. After delivery it blocks (pending_questions: "Does this meet your needs?") and waits for feedback:
- User says "looks good" → complete → retrospective → terminate
- User says "add pricing" → re-plan (add/modify nodes) → re-execute → deliver updated draft
- ~~No response for configurable timeout → auto-complete~~ **Amended by Amendment #2 (Decision #35) and Amendment #3 (Decision #36)**: Waits indefinitely. No auto-complete. User must explicitly approve.

This closes the ITERATE step in the task lifecycle (`INTAKE → DECOMPOSE → EXECUTE → CHECKPOINT → ITERATE → COMPLETE`). Same pending_questions mechanism handles: clarifying questions, problem escalation, AND iterative feedback. No new infrastructure.

### 22. Orchestrator uses direct `provider.chat()`, not `process_direct()`
The orchestrator makes raw LiteLLM calls for its own reasoning (workflow design, diagnosis, retrospective). Does NOT route through the agent loop. Eliminates concurrency races with other `process_direct()` callers. Nanobot informed via session notes — knows what happened, doesn't process orchestrator's internal reasoning.

**Workers**: Already isolated via `SubagentManager.spawn()` — no change.
**Heartbeat/Dream/Weekly**: Keep `process_direct()` (they need tools + session management) but fix races: local variables instead of `self.model` mutation, per-call tool context.

### 23. Provider resolution: LiteLLM native routing
Orchestrator dispatches workers by calling `litellm.acompletion(model="google/gemini-2.5-flash")`. LiteLLM routes to the correct provider using already-configured env vars (`GEMINI_API_KEY`, `ANTHROPIC_API_KEY`, etc.). Zero custom provider registry code needed. Resolves Open Question #1.

### 24. Tier restriction: split RouterProvider classify/chat
RouterProvider's `classify()` step exposed separately from `chat()`. After classification resolves the provider, agent loop checks: local provider (LM Studio connected) → exclude TaskTool. Cloud provider → include TaskTool. Works because LM Studio = Tier 0-2, cloud = Tier 3+. Lower-tier identity files include escalation prompt: "If you recognize the user wants substantial work done but you cannot create tasks, escalate." Amends Decision #4.

### 25. Orchestrator cancellation: token + task registry
`TaskWorker` maintains `_running_orchestrators: dict[str, asyncio.Task]`. `/cancel` (or natural language "cancel that task" → nanobot uses TaskTool cancel) sets a cancellation event AND calls `asyncio_task.cancel()`. Orchestrator checks `cancellation_event.is_set()` before each dispatch. Two-layer: graceful (event check) + hard kill (asyncio cancel).

### 26. Bootstrap workflow confirmation ~~(first 30 days)~~ (ALWAYS)
Orchestrator shows workflow design before executing: `[Task #3 — Orchestrator] I plan to do this in 3 steps: research → compare → synthesize. Budget ~$0.15. Proceed?` User approves or adjusts. ~~After 30 days, goes silent per normal policy.~~ **Amended by Amendment #3 (Decision #37)**: Always requires plan approval. No sunset. User explicitly removes this guard rail when ready. Soul file guidance in `orchestrator_soul.md`.

### 27. SqlitePool mandate
ALL database access goes through `SqlitePool` (existing at `copilot/db.py` — WAL mode, retry on SQLITE_BUSY, periodic checkpoint). Replace all `async with aiosqlite.connect(db_path)` patterns with pool access. Prevents write contention under concurrent orchestrator/worker/heartbeat/dream load.

### 28. Phase 0: Infrastructure wiring (pre-brain, testable now)
New pre-Phase 1 plan: ~10 tasks fixing critical plumbing in the CURRENT codebase before any brain architecture is built. Testable with nanobot's existing functionality. Scope: SqlitePool mandate, `process_direct()` race fix, `get_next_pending()` query fix, tool name→class registry, crash recovery for awaiting tasks, LiteLLM routing verification, RouterProvider classify/chat split, session_key passthrough, cancellation infrastructure, wake-after-resume events.

### 29. Bug fixes (delegated)
- `get_next_pending()`: Change to `status = 'pending'` only (not `IN ('pending', 'active')`). Orchestrator owns the full lifecycle once it picks up a task.
- Wake events after restart: Startup scan finds tasks in `awaiting` with empty `pending_questions` → set to `active` → orchestrator resumes from workflow state.
- `tools_whitelist`: Add `TOOL_REGISTRY: dict[str, type]` mapping string names to tool classes in SubagentManager. ~10 lines.
