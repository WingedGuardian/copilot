# ~~V2 Design: Task Lifecycle Engine (V2.1)~~

> **SUPERSEDED** by `2026-02-16-v2.1-brain-architecture-plan.md`. This is the old design doc. The brain architecture plan replaces the regex task detector and heuristic feedback handler with LLM-native approaches, and replaces flat sequential steps with a DAG-based orchestrator.

## Context

V1 built a reactive assistant — it answers when asked. V2's goal is an **autonomous digital executive assistant** that takes to-do items and gets them done. The user is the product owner; the system does 80%+ of the work. The primary interface is WhatsApp, interactions are asynchronous.

V2.1 is the first milestone: prove the task lifecycle works end-to-end. The proof-of-life is: user texts a research task from their phone, the system works on it asynchronously, and delivers a structured result via WhatsApp — without the user sitting at a desk.

Later milestones add coding agent integration (OpenCode), model pool redesign, browser automation, and external to-do system integration. But those all hang on the task lifecycle skeleton built here.

## Architecture: The Task Lifecycle

```
User message → INTAKE → DECOMPOSE → EXECUTE → CHECKPOINT → ITERATE → COMPLETE
                 │          │           │           │            │          │
              Detect if   Frontier    Workers    Report to    Incorporate  Present
              task vs     model       execute    user via     feedback,    deliverable
              chat        breaks      steps      WhatsApp     re-plan
                          into steps
```

**Model tiers** (formalized later in model pool design):
- **Thinker** (Opus-class): Decomposition, iteration, re-planning — executive reasoning
- **Coordinator** (Sonnet/Gemini-class): Task management, progress reporting, context resolution
- **Workers**: Specialized per step — web search, shell, subagent, future: coding agent, browser

### New pieces to build

#### 1. Task Detection Heuristic
**Where**: New function in `nanobot/copilot/routing/heuristics.py` or new file `nanobot/copilot/tasks/detector.py`
**What**: Classify incoming messages as "task request" vs "conversation." Heuristic-first (imperative verbs, action phrases, "can you...", "I need..."), with LLM fallback for ambiguous cases.
**Integration point**: Called in `nanobot/agent/loop.py` before the main agent loop. If task detected, create task entry and optionally enter clarification mode before decomposing.

#### 2. Decomposition Prompt + Wiring
**Where**: `nanobot/copilot/tasks/worker.py` — wire `decompose_fn` parameter
**What**: A prompt template sent to a frontier model that takes a task description + any clarifying answers and returns structured steps (JSON array of {description, tool_type, depends_on, estimated_effort}).
**Model**: Uses thinker tier (Opus-class) — this is the highest-leverage LLM call in the system.

#### 3. Step Execution Router
**Where**: `nanobot/copilot/tasks/worker.py` — wire `execute_fn` parameter
**What**: For each step, dispatch to the appropriate tool:
- `research` → subagent with web_search + web_fetch tools
- `shell` → ExecTool (extended timeout for long commands)
- `write` → subagent with filesystem tools
- `code` → (V2.2) OpenCode via shell
- `browse` → (V2.3) Playwright/Agent-S via shell
**Constraints**: Subagent iteration limit needs increase (15 → configurable, default 50 for task steps). Shell timeout needs extension for task-mode execution (60s → configurable, default 300s).

#### 4. Checkpoint + Progress Notification
**Where**: New logic in `nanobot/copilot/tasks/worker.py`
**What**: After each step completes (or when a step blocks), send a progress message via the message bus to the user's WhatsApp. Format:
```
Task #12 update:
✅ [completed steps]
⏳ [in-progress step]
❓ [questions/blockers]
```
**Integration**: Uses existing `_deliver_msg()` in `cli/commands.py` → message bus → WhatsApp channel.

#### 5. Feedback Loop
**Where**: New handler in agent loop or new file `nanobot/copilot/tasks/feedback.py`
**What**: When the user replies and there are pending task questions, the coordinator model determines:
- Is this a response to a task question? → Route answer to the blocked task, resume execution
- Is this a new task? → Create new task
- Is this just conversation? → Handle normally
**Context**: Coordinator sees recent task questions + user message. Natural language resolution (no structured format required).

#### 6. Task Commands
**Where**: Extend `nanobot/copilot/status/tool.py` or new command handler
**What**:
- `/tasks` — list all active tasks with status
- `/task <id>` — detailed status of a specific task
- `/cancel <id>` — cancel a task
These are slash commands handled in the agent loop, similar to existing `/status`, `/use`, `/private`.

## V2 Roadmap Context

This V2.1 task lifecycle engine is the skeleton for the full V2 vision:
- **V2.2**: Coding agent (OpenCode), model pool, parallel task execution
- **V2.3**: Browser automation, external to-do integration, proactive task suggestions
- **V2.4+**: Sub-agent specialization, self-improvement, content pipelines

The existing V2 Architecture doc's phases (proactive scheduler, tiered reviews, deeper lessons) become supporting features that make the task engine smarter over time — not the core product.

## Open Source Landscape (for future Plan B pivot)

If nanobot's architecture hits ceilings, these are candidates to bolt on:
- **[CrewAI](https://github.com/crewAIInc/crewAI)**: Role-based multi-agent orchestration (Python, production-ready)
- **[OpenCode](https://opencode.ai/)**: Model-flexible coding agent CLI (75+ models, 95K GitHub stars)
- **[Agent S](https://github.com/simular-ai/Agent-S)**: Computer-use agent for browser/GUI tasks
- **[MassGen](https://github.com/massgen/MassGen)**: Multi-agent parallel execution
