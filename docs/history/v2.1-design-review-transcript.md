### 10 Critique Points Identified

1. **Orchestrator LLM calls vs provider infrastructure** — Dedicated LiteLLMProvider bypasses RouterProvider's API keys, failover chains, and cost tracking. Orchestrator calls might fail silently or be invisible to cost analysis.

2. **SubagentManager announce fallback (MOST DANGEROUS)** — `_announce_result()` publishes raw worker output to the bus → agent loop → user. If `result_callback` fails, worker output leaks to WhatsApp. Same pattern as the approval system failure.

3. **"No truncation" result chaining** — Full worker output (10-20K tokens) piped to next node blows context windows for cheap models. Contradicts "cheap model for simple tasks" principle.

4. **60-second poll latency** — TaskWorker polls every 60s. Up to 60s dead time between task creation and anything happening. User thinks system is broken.

5. **Crash recovery without orchestrator context** — New orchestrator has no memory of original strategy. `design_reasoning` helps but isn't the full picture.

6. **Single-task bottleneck** — TaskWorker processes ONE task per tick. Multiple tasks execute sequentially, not in parallel. User expects parallel delegation.

7. **Lesson #2 contradicts Decision #1** — Heuristic routing (proven) vs LLM-based task detection (untested). Local models might not reliably distinguish task vs chat.

8. **SQLite write contention** — Multiple concurrent writers (orchestrator, workers, agent loop, dream cycle). Fix: WAL mode.

9. **Learning loop quality** — Dream cycle auto-updating identity files risks wrong conclusions from limited data. Need staging/approval, not auto-apply.

10. **Worker tools not restricted** — SubagentManager gives all tools regardless of orchestrator's assignment. `tools_whitelist` needed as creation filter, not runtime filter.

**User agreed**: Two most dangerous are #2 (bus announce fallback) and #4+#6 (poll latency + sequential tasks).

### Research conducted:
- [OpenClaw heartbeat](https://docs.openclaw.ai/gateway/heartbeat): HEARTBEAT.md file, HEARTBEAT_OK pattern, configurable intervals, visibility flags per channel

- [CHI 2025 "Inner Thoughts"](https://dl.acm.org/doi/10.1145/3706598.3713760): Two parallel tracks (overt communication + covert thought processing), inspired by SOAR/ACT-R cognitive architectures
- [Blackboard architecture revival](https://arxiv.org/html/2507.01701v1): LbMAS framework, shared memory, opportunity-driven agent activation

### Heartbeat design:
- Tier 0 SLM, programmatic checks + minimal LLM analysis
- Writes to blackboard (SQLite), never to user
- Living reference file: `data/copilot/heartbeat.md` (200-300 tokens, improved by dream/weekly review)
- Rotating roster of focus areas across ticks (tasks, conversation reflection, deadlines, anomalies, opportunities)
- Quality tied directly to learning hierarchy: weekly review → dream cycle → heartbeat reference

### Learning hierarchy established:
```
Weekly Review (Tier 4) → improves → Dream + Heartbeat
Dream Cycle (Tier 2/3) → improves → Heartbeat, reviews heartbeat quality
```

  system." True for reads. Problematic for writes.

  data. The dream cycle reads and writes. The TaskManager updates task status. All of these use async with
  aiosqlite.connect(db_path) — separate connections per call.

  SQLite supports concurrent readers but serializes writers. With aiosqlite, writes queue up. Under load (3-4 parallel
  workers completing simultaneously, plus the agent loop handling a message, plus the orchestrator evaluating results),
  you'll hit write contention. SQLite's default BUSY_TIMEOUT is 5 seconds — but aiosqlite wraps this in an executor
  thread, so the async pattern can mask the contention (the coroutine awaits while the thread blocks on the lock).

  What could go wrong: Under peak load, database writes queue up. If a worker's result write takes too long because
  other writes are pending, the orchestrator's asyncio.gather waits longer than expected. Not catastrophic, but adds
  unpredictable latency to a system that's supposed to feel responsive.

  Fix needed: Either (a) use WAL mode explicitly (PRAGMA journal_mode=WAL) which allows concurrent reads with one
  writer, (b) use a single shared connection for all writes (serialized by an async queue), or (c) just set a generous
  busy_timeout and accept the queueing. Option (a) is one line and should be standard.

  - orchestrator = the model that manages tasks (could be the same as big, could be Opus)

  Local model restrictions

  You're right this is a niche case, but it's the niche case that kills trust. Someone texts from their phone, the local

  - Creates a task it shouldn't (no TaskTool = solved)
  - Mishandles a message that should have been a task (needs escalation)

  escalation pattern cover the edge cases.

  - Fails stuck tasks (>30 min active with no progress)

  - Orchestrator crashes mid-execution → task stays "active", workflow has running nodes that will never complete

  1. Orchestrator liveness: Track orchestrator coroutines the way SubagentManager tracks subagents — last_activity

  2. Task-aware thresholds: Don't fail tasks at 30 min flat. Check the workflow — if workers are actively running

  3. Orphan detection: If a task is "active" but has no corresponding orchestrator coroutine in memory, it's orphaned.
  Mark it for crash recovery.

  Here's how I'd draw the line:

  Heartbeat (real-time health, every 2h):

  - Are there orphaned tasks? (crash recovery trigger)
  - Are workers stuck? (existing stuck subagent check)
  - Is cost tracking anomalous? (budget burn rate check — e.g., task spent 80% of budget in first node)

  Dream Cycle (batch analytics, daily):
  - Review completed task workflows: what worked, what didn't
  - Extract patterns from orchestrator retrospectives
  - Cross-reference model assignments with outcomes
  - Propose updates to identity files (agents.md, orchestrator_soul.md)
  - Cost efficiency analysis across all tasks in the past 24h
  - Identify recurring failure patterns

  Monitor (infrastructure, every 5 min):
  - Unchanged — subsystem health (Redis, Qdrant, SQLite, WhatsApp bridge)

  The key insight: the heartbeat is the immune system (detect and respond to problems in real-time), the dream cycle is

  3. The Heartbeat: Reframed as Proactive Peripheral Vision

  This is where it gets interesting. You're absolutely right — "health check" dilutes it. Let me reframe based on what
  you described, what I found in the research, and what I think could work.

  What the Heartbeat Actually Is

  The heartbeat is the system's peripheral vision — a cheap, frequent, background perception loop that scans the
  environment and updates a shared context (the "blackboard") that nanobot reads on its next interaction. It's what
  makes the system proactive in a world where the main LLM is purely reactive.

  The analogy to human cognition is precise: you don't consciously monitor your heartbeat, your breathing, the ambient
  temperature, or whether you're hungry. Your autonomic nervous system does that and only surfaces it to consciousness
  when something warrants attention. The heartbeat is that autonomic layer.

  The Blackboard

  The blackboard is the shared context between the heartbeat and nanobot. It's a structured set of observations — not
  prose, not prompts. Think of it as a news ticker that nanobot glances at before each interaction:

  BLACKBOARD (SQLite: blackboard_observations)
  ─────────────────────────────────────────────
  id          | timestamp        | category      | priority  | content                                          | ack
  obs_001     | 2026-02-17 14:00 | task_progress | info      | Task abc: research complete, synthesis running    | false
  obs_002     | 2026-02-17 14:00 | deadline      | attention | Car registration due in 3 days, no task created   | false
  obs_003     | 2026-02-17 14:00 | cost          | info      | Daily spend: $0.23 (normal)                       | false
  obs_004     | 2026-02-17 14:00 | opportunity   | info      | User mentioned "landing page" 2x this week, no task|
  false


  Nanobot sees these on its next interaction. Its LLM decides:
  - obs_001: Don't mention (user didn't ask, task is progressing fine)
  - obs_002: Mention casually ("By the way, your car registration is due in 3 days — want me to look into what's
  needed?")
  - obs_003: Don't mention (normal)
  - obs_004: Maybe mention if context is right ("You've mentioned a landing page a couple times — want me to start on
  that?")

  The LLM's judgment is what makes this elegant. Not every heartbeat observation becomes a message. Not every
  attention-level observation gets surfaced. The LLM knows the user, the context, the conversation, and decides.

  What the Heartbeat Does on Each Tick

  Programmatic checks (no LLM, zero cost):
  - Query task statuses from SQLite → any completions, failures, or stuck states?
  - Check orchestrator liveness → any coroutines idle longer than expected?
  - Check cost burn rate → any anomalies?
  - Check time-based triggers → any deadlines approaching? (from user profile, task metadata)
  - Check for undelivered messages → anything the orchestrator sent that wasn't acknowledged?

  SLM analysis (Tier 0, near-zero cost):
  - Review current blackboard state + recent changes
  - Any patterns worth noting? ("This is the third time a Gemini Flash worker failed on a research task this week")
  - Any context worth pre-assembling? ("User will probably ask about the VPS task soon — pre-fetch the results")
  - Any proactive observations? ("User mentioned dentist but never created a task")

  Write to blackboard:
  - Structured observations, categorized and prioritized
  - If nothing noteworthy: write nothing (or a minimal "all quiet" marker)
  - HEARTBEAT_OK equivalent: the heartbeat can be silent. Most ticks should be silent.

  What the Heartbeat Never Does

  - Never takes action. It observes. It doesn't cancel tasks, restart workers, or modify state.
  - Never talks to the user. It writes to the blackboard. Nanobot decides what to surface.
  - Never second-guesses the orchestrator. It can observe "orchestrator has been running for 45 minutes" but it doesn't
  decide whether that's a problem — it logs the observation and lets nanobot's LLM (or you) decide.
  - Never uses cloud models. Tier 0 only. It's designed to be cheap and frequent.

  How This Makes the System Proactive

  This is the key insight you identified: the main LLM is purely reactive (it processes your messages). The heartbeat is
   what gives the system initiative.

  Without the heartbeat, nanobot only thinks when you text it. With the heartbeat, nanobot thinks about your world
  continuously, and when you DO text it, it already has context about what's been happening:

  - "Your VPS comparison task finished 20 minutes ago — here's the result" (instead of you having to ask)
  - "By the way, I noticed the domain you asked about last week expires in 5 days" (proactive)
  - "Three tasks completed overnight — want a summary?" (morning awareness)

  The https://dl.acm.org/doi/10.1145/3706598.3713760 describes exactly this pattern: a proactive agent running two
  parallel tracks — overt communication and covert thought processing. The heartbeat IS the covert thought track.

  https://github.com/NevaMind-AI/memU calls this "continuous sync" — their background agent monitors, memorizes,
  predicts, and pre-fetches context. https://docs.openclaw.ai/gateway/heartbeat implements it as a workspace-file-driven
   checklist (HEARTBEAT.md) that runs every 30 minutes with a HEARTBEAT_OK no-op pattern when nothing needs attention.

  Where I'd Push This Further

  The existing heartbeat code (dream/heartbeat.py) is 90% programmatic health checks with a bolted-on LLM call for task
  review. The reframe means:

  1. Rename it conceptually — in the code, in the docs, and in nanobot's identity files. It's not
  CopilotHeartbeatService doing "health checks." It's the system's Autonomic Layer — its ambient awareness.
  2. The blackboard becomes the integration point. The heartbeat writes observations. The orchestrator writes task
  progress. The dream cycle writes analytical findings. External signals (calendar, email, webhooks) write events.
  Nanobot reads the blackboard on every interaction and has full situational awareness.
  3. The existing heartbeat_events table and get_unacknowledged_events() injection in loop.py is already 80% of the
  blackboard. The code path exists — heartbeat writes events, agent loop injects unacknowledged events into context. The
   reframe is: expand what gets written there (not just health alerts, but observations, opportunities, context), and
  make the injection smarter (let the LLM see the full blackboard, not just "alerts").
  4. The HEARTBEAT.md pattern from OpenClaw is worth stealing. A workspace file that defines what the heartbeat should
  pay attention to. The user (or nanobot) can modify it over time. "Check on my running tasks. Note approaching
  deadlines. Watch for cost anomalies." This makes the heartbeat's attention configurable without code changes.

  The Health Check Stuff Is Still Necessary — Just Secondary

  You're right that you don't want a system that breaks and crashes constantly. But the health checks (Redis alive?
  Qdrant reachable? SQLite locked?) should be programmatic, not LLM-mediated. A ping is a ping. No SLM needed to check
  if Redis returns PONG.

  The health checks are the heartbeat's baseline — the things it checks BEFORE it even considers doing SLM analysis. If
  the infrastructure is broken, that's an urgent observation on the blackboard and nanobot should tell you immediately.
  But the interesting, valuable, proactive part of the heartbeat is everything ABOVE that baseline: the pattern
  detection, the opportunity spotting, the context pre-assembly.

  Your instinct is good but I think the execution needs to be different from what you described. Here's why and what I'd
   suggest instead:

  Your idea: heartbeat asks nanobot "what could you have said differently in the last X exchanges?"

  The problem you identified: this makes every heartbeat a mandatory nanobot LLM call, which is expensive and defeats
  "nanobot is purely reactive."

  My counter-proposal: The heartbeat does the reflection ITSELF, with the SLM. The SLM reads the last few exchanges
  (from session history in JSONL, which it can access) and runs a focused analysis. If — and ONLY if — the SLM finds
  something worth flagging, it writes an observation to the blackboard. Nanobot sees it on the next interaction.

  What the SLM can realistically catch at Tier 0 (4B):
  - A direct question that wasn't answered (pattern-matchable)
  - A topic that was dropped mid-conversation (detectable)
  - A commitment nanobot made ("I'll look into that") with no follow-up task
  - A factual claim without citation when the topic warranted it

  What it CAN'T catch (leave for dream cycle):
  - Subtle tone mismatches
  - Strategic misses ("you should have asked about budget before suggesting options")
  - Quality of reasoning in complex answers

  This keeps the heartbeat cheap (SLM-only) while providing genuine introspective value. The dream cycle does the deeper
   reflection (Tier 2/3 can actually evaluate reasoning quality).

  Instead of checking everything every tick (expensive, unfocused), the heartbeat rotates through attention areas:

  ┌──────┬─────────────────────────┬─────────────────────────────────────────────────────────────┐

  ├──────┼─────────────────────────┼─────────────────────────────────────────────────────────────┤
  │ 4    │ Anomaly detection       │ Unusual cost patterns? Repeated errors?                     │
  ├──────┼─────────────────────────┼─────────────────────────────────────────────────────────────┤
  │ 5    │ Opportunity spotting    │ Recurring themes without tasks? Connections between topics? │
  ├──────┼─────────────────────────┼─────────────────────────────────────────────────────────────┤
  │ 6    │ Back to 1               │                                                             │
  └──────┴─────────────────────────┴─────────────────────────────────────────────────────────────┘

  Each tick is cheap because it's focused on ONE area. The reference file (heartbeat.md) defines the roster and what
  "significant" means for each area. The roster itself can be refined by dream/weekly review.

## Phase 7: Forward-Looking Failure Analysis (2026-02-17, continued session)

**User request**: "Lets think one step ahead too — what could go wrong not just on day 1, but halfway through? We need to think about this from end to end. Use AskUserQuestion to surface questions that need decisions."

### Analysis Approach
Explored the full codebase (cost/db.py, dream/cycle.py, dream/monitor.py, tasks/worker.py, tasks/manager.py, agent/subagent.py, copilot/config.py) for mid-lifecycle failure modes. Identified 20 failure modes across 5 categories:

**Data Accumulation & Decay (4 modes)**: Identity file bloat without token budgets, SQLite table growth, session note accumulation, stale blackboard observations.

**Learning Loop Pathologies (4 modes)**: Wrong conclusions from small samples (first 30 days most dangerous), circular reinforcement between dream cycle and identity files, lesson/observation conflict between tiers, auto-approval applying bad patterns.

**Operational Degradation (5 modes)**: Orchestrator liveness blind spots, worker zombie processes, cost creep from learning loop overhead, context window pressure from growing identity files, heartbeat SLM quality degradation over time.

**Trust & UX Erosion (4 modes)**: Message flood during AFK periods, notification fatigue, dream cycle surfacing irrelevant observations, loss of user confidence from false positives.

**Concurrency & State (4 modes)**: SQLite write contention (addressed by WAL mode), race conditions between dream cycle and orchestrator writing to same tables, stale context after compaction, orphaned workflow nodes.

### Three Highest-Priority Clusters
1. **Identity file entropy** — no token budget, no deduplication, no coherence audit, cost creep as files grow
2. **Wrong lessons from small samples** — no confidence gating, auto-approval doesn't filter quality, first 30 days most dangerous
3. **Orchestrator liveness + worker zombies** — deferred to pre-2.2, cancel_stuck() has no caller, flat 30-min threshold too blunt

### Decisions Made (via AskUserQuestion)

**Q1: Identity file token budget?**
→ **1000-token soft cap**. Weekly review (Tier 4) consolidates when exceeded. Evicted content moves to long-term memory (vector store). Identity files retain pointer keywords (`<!-- memory: keyword1, keyword2 -->`) so higher-tier models follow breadcrumbs into memory for depth. SLM reads compact version. First 30 days: all changes require user approval.

**Key insight from user**: Identity files become a curated index with pointers, not the full knowledge store. Long-term memory holds the overflow. The system's accumulated knowledge grows without bound while the working set stays SLM-friendly.

**Q2: Dream observation quality gate?**
→ **Weekly review surfaces, user approves**. Nothing auto-applies. Weekly review resurfaces relevant pending observations. First 30 days: fully manual. After 30 days: revisit whether to relax the gate. Each observation tracks `supporting_event_count`.

**Q3: SQLite data retention?**
→ **Store forever**. User challenged the premise: "Why can't we just store this forever?" Math showed ~30MB/year total across all tables. Not a real problem. More data makes learning loop smarter. Dropped as a failure mode — premature optimization concern.

**Q4: AFK message batching?**
→ **User-configurable** via `/settings notifications [immediate|digest|priority]`. Default: digest. Most flexible approach.
